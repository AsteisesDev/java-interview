## Из каких основных компонентов состоит персональный компьютер?

Персональный компьютер состоит из следующих ключевых аппаратных компонентов:

**Центральный процессор (CPU, Central Processing Unit)** — «мозг» компьютера. Выполняет арифметические, логические операции и управляет работой всех остальных компонентов. Современные процессоры содержат несколько ядер и работают на тактовых частотах порядка нескольких гигагерц.

**Оперативная память (RAM, Random Access Memory)** — энергозависимая память для хранения данных и команд, с которыми процессор работает в текущий момент. Обеспечивает быстрый произвольный доступ к данным (порядка наносекунд). При выключении компьютера содержимое RAM теряется.

**Накопители данных (HDD/SSD)** — энергонезависимая память для долговременного хранения данных. HDD (Hard Disk Drive) — жёсткий диск с вращающимися магнитными пластинами. SSD (Solid State Drive) — твердотельный накопитель на основе флеш-памяти, значительно более быстрый.

**Материнская плата (Motherboard)** — основная печатная плата, которая объединяет все компоненты компьютера. На ней расположены:
+ Сокет для процессора
+ Слоты для оперативной памяти (DIMM)
+ Слоты расширения (PCIe) для видеокарт, сетевых карт и т.д.
+ Чипсет — набор микросхем, управляющих обменом данных между процессором, памятью и периферией
+ Разъёмы для накопителей (SATA, M.2)
+ Порты ввода/вывода (USB, Ethernet, аудио)

**Шины (Bus)** — коммуникационные каналы для передачи данных между компонентами:
+ **Шина данных** — передаёт собственно данные между процессором, памятью и устройствами
+ **Адресная шина** — передаёт адреса ячеек памяти или портов ввода/вывода
+ **Шина управления** — передаёт управляющие сигналы (чтение, запись, прерывания)
+ **PCIe (PCI Express)** — современная последовательная высокоскоростная шина для подключения видеокарт, SSD (NVMe), сетевых карт
+ **SATA** — шина для подключения накопителей (HDD, SSD)

**Блок питания (PSU)** — преобразует переменный ток из электрической сети в постоянный ток нужного напряжения (3.3В, 5В, 12В) для питания всех компонентов.

**Видеокарта (GPU)** — выполняет обработку и вывод графики. Может быть встроенной в процессор (iGPU) или дискретной (отдельная плата). GPU также активно используется для параллельных вычислений (GPGPU), машинного обучения.

## Что такое процессор (CPU)? Что такое ядра, потоки, тактовая частота?

**CPU (Central Processing Unit)** — центральный процессор, основной вычислительный компонент компьютера, выполняющий машинные инструкции.

**Ядро (Core)** — независимый вычислительный блок внутри процессора. Каждое ядро имеет собственные исполнительные блоки (ALU, FPU), регистры и, как правило, собственный кэш L1 и L2. Многоядерные процессоры (2, 4, 8, 16 и более ядер) могут выполнять несколько потоков одновременно, параллельно.

**Поток (Thread / Аппаратный поток)** — логический вычислительный поток. Технология Hyper-Threading (Intel) / SMT (AMD) позволяет одному физическому ядру обрабатывать два потока одновременно. Это достигается за счёт дублирования некоторых блоков ядра (регистров, очередей), в то время как исполнительные блоки (ALU) разделяются. Например, процессор с 8 ядрами и Hyper-Threading имеет 16 аппаратных потоков. Это значит, что ОС видит 16 логических процессоров.

**Тактовая частота (Clock Speed)** — количество тактов (циклов) в секунду, измеряется в гигагерцах (ГГц). За каждый такт процессор выполняет определённую часть работы (стадию конвейера). Например, 4.0 ГГц означает 4 миллиарда тактов в секунду. Более высокая тактовая частота, как правило, означает более высокую производительность на одно ядро, но это зависит также от архитектуры (IPC — Instructions Per Cycle).

**Разрядность** — определяет размер данных, обрабатываемых за одну операцию. Современные процессоры 64-битные, то есть за один раз работают с 64-битными числами и адресами, что позволяет адресовать более 4 ГБ оперативной памяти (ограничение 32-битных систем).

**Для Java-разработчика** это важно, потому что:
+ Количество ядер определяет реальный параллелизм при работе с потоками Java (`Thread`, `ExecutorService`)
+ JVM использует аппаратные возможности процессора: JIT-компилятор оптимизирует код под конкретную архитектуру
+ Тактовая частота влияет на скорость однопоточных операций (например, GC pause)

## Что такое кэш процессора (L1, L2, L3)?

**Кэш процессора** — это сверхбыстрая память малого объёма, расположенная непосредственно на кристалле процессора. Кэш хранит копии часто используемых данных из оперативной памяти, чтобы процессору не приходилось каждый раз обращаться к более медленной RAM.

Кэш организован в несколько уровней (иерархия):

| Уровень | Объём (типично) | Задержка (латентность) | Описание |
|---------|-----------------|------------------------|----------|
| **L1** | 32–128 КБ на ядро | ~1–4 такта (~1 нс) | Самый быстрый и маленький. Разделён на L1d (данные) и L1i (инструкции). Индивидуальный для каждого ядра |
| **L2** | 256 КБ – 1 МБ на ядро | ~4–14 тактов (~3–5 нс) | Быстрый, средний объём. Обычно индивидуальный для каждого ядра |
| **L3** | 8–64 МБ (общий) | ~30–70 тактов (~10–20 нс) | Самый большой, но медленнее. Общий для всех ядер процессора |
| **RAM** | 8–128 ГБ | ~200–300 тактов (~50–100 нс) | Основная оперативная память, значительно медленнее кэша |

**Принцип работы кэша:**
1. Процессор запрашивает данные по адресу
2. Сначала проверяется L1. Если данные найдены — это **cache hit** (попадание)
3. Если нет (cache miss) — проверяется L2, затем L3, и только потом — RAM
4. При загрузке данных из RAM они кэшируются на всех уровнях

**Принцип локальности** — основа эффективной работы кэша:
+ **Временна́я локальность** — если данные были использованы, они, скорее всего, понадобятся снова
+ **Пространственная локальность** — если данные по адресу N использованы, скоро понадобятся данные по адресам N+1, N+2 и т.д.

**Кэш-линия (cache line)** — минимальная единица данных, загружаемая в кэш. Обычно 64 байта. Даже если нужен 1 байт, загружается целая кэш-линия.

**Для Java-разработчика:**
+ Обход массива последовательно (`array[0], array[1], ...`) значительно быстрее, чем случайный доступ к `LinkedList`, из-за пространственной локальности
+ **False sharing** — когда два потока модифицируют разные переменные, попавшие в одну кэш-линию. Это приводит к постоянной инвалидации кэша и серьёзной деградации производительности. В Java можно использовать аннотацию `@Contended` (или padding) для борьбы с этим

## Как CPU выполняет инструкции? Опишите цикл fetch-decode-execute.

Процессор выполняет программу, последовательно проходя через цикл **fetch-decode-execute** (выборка — декодирование — выполнение) для каждой инструкции:

**1. Fetch (Выборка)**
+ Процессор считывает следующую инструкцию из памяти по адресу, хранящемуся в **Program Counter (PC)** / **Instruction Pointer (IP)**
+ Инструкция загружается из кэша L1i (или из RAM, если её нет в кэше) в **регистр инструкции (IR)**
+ PC увеличивается, чтобы указывать на следующую инструкцию

**2. Decode (Декодирование)**
+ Блок декодирования разбирает инструкцию: определяет тип операции (сложение, загрузка из памяти, переход и т.д.), операнды (регистры, адреса памяти, константы)
+ В современных процессорах (x86) сложные CISC-инструкции декодируются в более простые внутренние микрооперации (μops)

**3. Execute (Выполнение)**
+ Инструкция отправляется в соответствующий исполнительный блок:
  + **ALU (Arithmetic Logic Unit)** — для арифметических и логических операций
  + **FPU (Floating Point Unit)** — для операций с числами с плавающей запятой
  + **AGU (Address Generation Unit)** — для вычисления адресов памяти
+ Выполняется операция над данными

**4. Memory Access (Доступ к памяти)** — если инструкция работает с памятью (load/store), происходит чтение или запись данных через иерархию кэшей.

**5. Write-back (Запись результата)** — результат записывается обратно в регистр или в память.

В действительности современные процессоры не выполняют инструкции строго по одной. Они используют:
+ **Конвейеризацию (pipeline)** — разные стадии разных инструкций выполняются одновременно
+ **Суперскалярность** — несколько инструкций выполняются параллельно на разных исполнительных блоках
+ **Out-of-order execution** — инструкции выполняются не в порядке программы, а по мере готовности данных, что повышает загрузку исполнительных блоков

## Что такое конвейер процессора (pipeline)?

**Конвейер (pipeline)** — это техника организации работы процессора, при которой выполнение инструкции разбивается на несколько стадий, и разные стадии разных инструкций выполняются одновременно.

По аналогии с конвейером на заводе: пока одна деталь проходит покраску, предыдущая уже сушится, а следующая режется.

**Пример простого 5-стадийного конвейера:**

```
Такт:       1    2    3    4    5    6    7    8
Инстр. 1: [IF] [ID] [EX] [MEM][WB]
Инстр. 2:      [IF] [ID] [EX] [MEM][WB]
Инстр. 3:           [IF] [ID] [EX] [MEM][WB]
Инстр. 4:                [IF] [ID] [EX] [MEM][WB]
```

Где:
+ **IF** — Instruction Fetch (выборка инструкции)
+ **ID** — Instruction Decode (декодирование)
+ **EX** — Execute (выполнение)
+ **MEM** — Memory Access (доступ к памяти)
+ **WB** — Write Back (запись результата)

Без конвейера 4 инструкции выполняются за 20 тактов (5 × 4). С конвейером — за 8 тактов. В идеале после заполнения конвейера каждый такт завершается одна инструкция.

**Проблемы конвейера (hazards):**

+ **Конфликт данных (data hazard)** — следующая инструкция зависит от результата предыдущей, которая ещё не завершилась. Решается через **forwarding** (пробрасывание результата до стадии WB) или вставку «пузырей» (stall)
+ **Конфликт управления (control hazard)** — условный переход (if/else): процессор не знает, какую инструкцию загружать следующей. Решается через **предсказание переходов (branch prediction)**: процессор предполагает, куда пойдёт переход. Если ошибся — конвейер сбрасывается (pipeline flush), что стоит десятков тактов
+ **Структурный конфликт (structural hazard)** — два действия одновременно требуют один и тот же ресурс

Современные процессоры имеют очень глубокие конвейеры (14–20+ стадий) и сложные предсказатели переходов с точностью свыше 95%.

## Как работает оперативная память (RAM)?

**RAM (Random Access Memory)** — оперативная память с произвольным доступом. Это значит, что время доступа к любой ячейке памяти одинаково, в отличие от последовательных носителей.

**Тип: DRAM (Dynamic RAM)** — основной тип оперативной памяти в ПК. Каждый бит хранится как заряд конденсатора. Конденсаторы со временем теряют заряд, поэтому DRAM требует периодической регенерации (refresh) — отсюда «динамическая».

**Стандарты DDR (Double Data Rate):**

| Стандарт | Частота (эффективная) | Пропускная способность (на канал) | Напряжение |
|----------|-----------------------|-----------------------------------|------------|
| DDR3 | 800–2133 МГц | до ~17 ГБ/с | 1.5В |
| DDR4 | 2133–5100 МГц | до ~40 ГБ/с | 1.2В |
| DDR5 | 4800–8400+ МГц | до ~67 ГБ/с | 1.1В |

DDR передаёт данные по обоим фронтам тактового сигнала (восходящему и нисходящему), удваивая скорость.

**Адресация памяти:**
+ Память организована как массив ячеек, каждая имеет уникальный адрес
+ В 32-битных системах адрес — 32 бита, что позволяет адресовать максимум 2³² = 4 ГБ
+ В 64-битных системах теоретический максимум — 2⁶⁴ байт (16 эксабайт), но на практике используется 48 бит (256 ТБ)
+ **Байтовая адресация** — каждый байт имеет свой адрес

**Многоканальность (Dual/Quad Channel):**
+ Использование двух или четырёх модулей памяти одновременно удваивает/учетверяет пропускную способность
+ Для работы многоканальности модули должны быть установлены в соответствующие слоты

**Для Java-разработчика:**
+ JVM выделяет heap в оперативной памяти. Размер heap ограничен параметрами `-Xms` и `-Xmx`
+ Объём RAM определяет, сколько данных приложение может держать в памяти. Если памяти не хватает, ОС начинает использовать swap (подкачку), что катастрофически снижает производительность
+ Доступ к RAM (~100 нс) на порядки медленнее кэша CPU, но на порядки быстрее диска (SSD ~100 мкс, HDD ~10 мс)

## Что такое виртуальная память?

**Виртуальная память** — это механизм управления памятью, при котором каждый процесс работает со своим собственным виртуальным адресным пространством, которое отображается на физическую оперативную память (и диск) средствами ОС и аппаратного обеспечения (MMU).

**Зачем нужна виртуальная память:**
+ **Изоляция процессов** — каждый процесс видит только свою память и не может повредить память другого процесса
+ **Абстракция** — процесс думает, что у него есть непрерывное адресное пространство, хотя в физической памяти данные могут быть разбросаны
+ **Расширение памяти** — суммарный объём виртуальной памяти всех процессов может превышать объём физической RAM за счёт использования диска (swap)

**Страничная организация (Paging):**
+ Виртуальное адресное пространство разбивается на **страницы (pages)** фиксированного размера (обычно 4 КБ)
+ Физическая память разбивается на **фреймы (frames)** того же размера
+ **Таблица страниц (page table)** — структура данных, поддерживаемая ОС, которая хранит соответствие: виртуальная страница → физический фрейм
+ Каждый процесс имеет собственную таблицу страниц

**Трансляция адреса:**
1. Процесс обращается к виртуальному адресу
2. MMU (Memory Management Unit) — аппаратный блок в процессоре — разделяет адрес на номер страницы и смещение внутри страницы
3. По номеру страницы ищется физический фрейм в таблице страниц
4. Физический адрес = адрес фрейма + смещение

**TLB (Translation Lookaside Buffer):**
+ Специальный кэш в процессоре для хранения недавних трансляций «виртуальная страница → физический фрейм»
+ Обращение к TLB — 1–2 такта. Промах по TLB (TLB miss) требует обхода таблицы страниц — десятки тактов
+ TLB обычно содержит 64–1024 записи

**Подкачка (Swap):**
+ Когда физической памяти не хватает, ОС выгружает редко используемые страницы на диск (в swap-файл или swap-раздел)
+ Когда процесс обращается к выгруженной странице, возникает **page fault** (страничная ошибка): ОС приостанавливает процесс, загружает страницу с диска обратно в RAM, обновляет таблицу страниц и возобновляет процесс
+ Page fault — очень дорогая операция (миллисекунды), поэтому частое использование swap резко снижает производительность

**Для Java-разработчика:**
+ JVM-процесс работает в виртуальном адресном пространстве. Heap, стеки потоков, metaspace — всё это виртуальные адреса
+ Если JVM выделяет больше памяти, чем есть физической RAM, ОС начнёт swap, и GC-паузы могут вырасти с миллисекунд до секунд

## Чем отличаются HDD и SSD?

**HDD (Hard Disk Drive)** — жёсткий диск с механическими компонентами. Данные хранятся на вращающихся магнитных пластинах. Считывающая головка перемещается по пластине для чтения/записи.

**SSD (Solid State Drive)** — твердотельный накопитель без движущихся частей. Данные хранятся в ячейках флеш-памяти (NAND).

| Характеристика | HDD | SSD |
|---|---|---|
| Скорость последовательного чтения | 100–200 МБ/с | 500–7000 МБ/с (NVMe) |
| Скорость случайного доступа | ~0.1–1 IOPS/мс (5–10 мс) | ~0.05–0.1 мс |
| IOPS (случайное чтение 4K) | 75–200 | 50 000–1 000 000+ |
| Энергопотребление | Выше | Ниже |
| Шум / вибрации | Есть | Нет |
| Устойчивость к ударам | Низкая | Высокая |
| Цена за ГБ | Ниже | Выше |
| Объём | До 20+ ТБ | До 8+ ТБ |
| Срок службы | Ограничен механическим износом | Ограничен числом циклов записи (TBW) |

**Последовательный vs случайный доступ:**
+ **Последовательный доступ (sequential)** — чтение/запись данных, расположенных подряд. HDD справляется относительно хорошо, так как головка не перемещается
+ **Случайный доступ (random)** — чтение/запись данных в разных местах. На HDD это крайне медленно из-за перемещения головки (seek time ~5–10 мс). На SSD разница между последовательным и случайным доступом значительно меньше

**IOPS (Input/Output Operations Per Second)** — количество операций ввода/вывода в секунду. Ключевая метрика для оценки производительности накопителей при случайном доступе. Базы данных, как правило, создают нагрузку именно с случайным доступом.

**Интерфейсы подключения:**
+ **SATA** — до 600 МБ/с (SATA III). Используется для HDD и бюджетных SSD
+ **NVMe (через PCIe)** — до 7000+ МБ/с (PCIe 4.0 x4). Используется для высокопроизводительных SSD

**Для Java-разработчика:**
+ Тип накопителя критически важен для производительности баз данных (PostgreSQL, Elasticsearch), логирования и работы с файлами
+ SSD радикально ускоряют запуск приложений, сборку проектов (Maven/Gradle) и работу IDE

## Какие системы счисления используются в компьютерах?

В компьютерах используются три основные системы счисления:

**Двоичная (Binary, base-2)**
+ Цифры: `0` и `1`
+ Фундаментальная система для компьютеров, так как электронные компоненты имеют два состояния (есть ток / нет тока)
+ Префикс в Java: `0b` — например, `0b1010` = 10
+ 1 бит = одна двоичная цифра, 1 байт = 8 бит

Примеры:
```
Двоичное    Десятичное
0b0001      1
0b0010      2
0b0100      4
0b1000      8
0b1010      10
0b11111111  255
```

**Шестнадцатеричная (Hexadecimal, base-16)**
+ Цифры: `0–9` и `A–F` (A=10, B=11, ..., F=15)
+ Компактная запись двоичных данных: каждая hex-цифра = 4 бита (полубайт / nibble)
+ Префикс в Java: `0x` — например, `0xFF` = 255
+ Широко используется для: адресов памяти, цветов в CSS (`#FF0000`), MAC-адресов, UUID

Примеры:
```
Hex     Двоичное        Десятичное
0x0A    0000 1010       10
0xFF    1111 1111       255
0x1F4   0001 1111 0100  500
```

**Десятичная (Decimal, base-10)** — привычная человеку система, используется в исходном коде и выводе для пользователя.

**Перевод между системами в Java:**
```java
int x = 42;
Integer.toBinaryString(x);      // "101010"
Integer.toHexString(x);         // "2a"
Integer.toOctalString(x);       // "52"
Integer.parseInt("101010", 2);  // 42
Integer.parseInt("2a", 16);     // 42
```

**Представление отрицательных чисел — дополнительный код (two's complement):**
+ Старший бит — знаковый (0 = положительное, 1 = отрицательное)
+ Для получения отрицательного числа: инвертировать все биты и прибавить 1
+ Пример (8 бит): -1 = `1111 1111`, -128 = `1000 0000`
+ Для `int` (32 бита) в Java: диапазон от -2³¹ до 2³¹−1

## Что такое побитовые операции?

**Побитовые (битовые) операции** — операции, выполняемые над отдельными битами числа. В Java они применяются к целочисленным типам (`int`, `long`, `byte`, `short`).

**Основные побитовые операции:**

| Операция | Оператор (Java) | Описание | Пример (8 бит) |
|----------|-----------------|----------|-----------------|
| AND (И) | `&` | 1, только если оба бита = 1 | `1010 & 1100 = 1000` |
| OR (ИЛИ) | `\|` | 1, если хотя бы один бит = 1 | `1010 \| 1100 = 1110` |
| XOR (Исключающее ИЛИ) | `^` | 1, если биты различаются | `1010 ^ 1100 = 0110` |
| NOT (НЕ) | `~` | Инвертирует все биты | `~1010 = 0101` |
| Сдвиг влево | `<<` | Сдвигает биты влево, заполняя нулями | `0001 << 3 = 1000` |
| Сдвиг вправо (арифметический) | `>>` | Сдвигает вправо, сохраняя знаковый бит | `1000 >> 2 = 1110` (для отрицательных) |
| Сдвиг вправо (логический) | `>>>` | Сдвигает вправо, заполняя нулями | `1000 >>> 2 = 0010` |

**Практическое применение:**

```java
// Проверка чётности числа
boolean isEven = (n & 1) == 0;

// Установка бита в позиции pos
int result = value | (1 << pos);

// Сброс бита в позиции pos
int result = value & ~(1 << pos);

// Проверка бита в позиции pos
boolean isSet = (value & (1 << pos)) != 0;

// Умножение/деление на степень двойки
int mul8 = n << 3;   // n * 8
int div4 = n >> 2;   // n / 4

// Обмен значений без временной переменной (XOR swap)
a ^= b; b ^= a; a ^= b;

// Маски в EnumSet, правах доступа (rwx), битовых флагах
int READ = 1;    // 001
int WRITE = 2;   // 010
int EXEC = 4;    // 100
int perms = READ | EXEC; // 101 — чтение и выполнение
```

**Применение в Java:**
+ `java.util.EnumSet` — внутри использует битовую маску
+ `HashMap` — вычисление индекса бакета: `(n - 1) & hash`
+ Сетевые маски (IP): `255.255.255.0` = `11111111.11111111.11111111.00000000`

## Что такое кодировки символов? Чем отличаются ASCII, UTF-8, UTF-16?

**Кодировка символов** — это набор правил, определяющих, как символы (буквы, цифры, знаки) представляются в виде последовательности байтов в памяти компьютера.

**ASCII (American Standard Code for Information Interchange):**
+ 7-битная кодировка, 128 символов (0–127)
+ Включает: латинские буквы (A–Z, a–z), цифры (0–9), знаки препинания, управляющие символы (перенос строки, табуляция)
+ 1 символ = 1 байт
+ Не поддерживает кириллицу и большинство других алфавитов

**Unicode:**
+ Универсальный стандарт, присваивающий уникальный номер (**code point**) каждому символу всех языков мира
+ Содержит более 150 000 символов
+ Code point записывается как `U+XXXX`, например: `U+0041` = 'A', `U+0410` = 'А' (кирилл.), `U+1F600` = 😀
+ Unicode — это **не кодировка**, а набор символов. Кодировки UTF-8, UTF-16, UTF-32 определяют, как code points хранятся в байтах

**UTF-8 (Unicode Transformation Format, 8-bit):**
+ Кодировка переменной длины: 1–4 байта на символ
+ Совместима с ASCII: символы 0–127 занимают 1 байт (те же значения, что в ASCII)
+ Латиница — 1 байт, кириллица — 2 байта, иероглифы — 3 байта, эмодзи — 4 байта
+ Самая распространённая кодировка в Интернете (>98% веб-страниц)
+ Используется по умолчанию в Linux, JSON, XML, HTML5

**UTF-16:**
+ Кодировка переменной длины: 2 или 4 байта на символ
+ Большинство символов (включая кириллицу, латиницу, иероглифы) — 2 байта
+ Символы за пределами BMP (Basic Multilingual Plane), например эмодзи — 4 байта (суррогатная пара)
+ **Используется внутри Java**: `String` и `char` основаны на UTF-16. `char` = 2 байта = один элемент UTF-16

**Сравнение:**

| Свойство | ASCII | UTF-8 | UTF-16 |
|----------|-------|-------|--------|
| Размер символа | 1 байт | 1–4 байта | 2–4 байта |
| Совместимость с ASCII | Да | Да | Нет |
| Поддержка всех символов Unicode | Нет | Да | Да |
| Эффективность для латиницы | Максимальная | Максимальная | Низкая (2 байта на символ) |
| Эффективность для иероглифов | — | 3 байта | 2 байта |
| Использование | Устаревший стандарт | Веб, Linux, JSON, XML | Java, Windows, .NET |

**В Java:**
```java
String s = "Привет";
s.length();                    // 6 — количество char (UTF-16 единиц)
s.codePointCount(0, s.length()); // 6 — количество Unicode символов (code points)
s.getBytes("UTF-8").length;    // 12 — 6 кириллических символов × 2 байта UTF-8
s.getBytes("UTF-16").length;   // 14 — 6 × 2 + 2 (BOM)
```

Начиная с Java 9, строки внутри JVM могут храниться в **Compact Strings**: если все символы входят в Latin-1 (1 байт), используется `byte[]` вместо `char[]`, экономя память вдвое для ASCII-строк.

## Что делает операционная система? Каковы её основные функции?

**Операционная система (ОС)** — это системное программное обеспечение, которое управляет аппаратными ресурсами компьютера и предоставляет сервисы для прикладных программ. ОС является посредником между аппаратурой и пользовательскими приложениями.

**Основные функции ОС:**

**1. Управление процессами:**
+ Создание, выполнение и завершение процессов
+ Планирование (scheduling) — распределение процессорного времени между процессами
+ Механизмы синхронизации (мьютексы, семафоры)
+ Межпроцессное взаимодействие (IPC)

**2. Управление памятью:**
+ Выделение и освобождение оперативной памяти для процессов
+ Виртуальная память и трансляция адресов
+ Защита памяти — процесс не может обратиться к памяти другого процесса
+ Подкачка (swap) — использование диска при нехватке RAM

**3. Управление файловой системой:**
+ Организация данных на накопителях в виде файлов и каталогов
+ Операции: создание, чтение, запись, удаление, переименование
+ Управление правами доступа (rwx в Linux, ACL в Windows)
+ Поддержка различных файловых систем (ext4, NTFS, FAT32)

**4. Управление устройствами ввода/вывода:**
+ Абстрагирование работы с аппаратурой через драйверы
+ Унифицированный интерфейс работы с устройствами (в Unix — «всё есть файл»)
+ Буферизация, кэширование, спулинг (очередь печати)

**5. Управление сетью:**
+ Реализация сетевых протоколов (TCP/IP стек)
+ Сокеты — API для сетевого взаимодействия
+ Управление сетевыми интерфейсами

**6. Безопасность и контроль доступа:**
+ Аутентификация пользователей (логин/пароль)
+ Разграничение прав доступа к файлам и ресурсам
+ Изоляция процессов друг от друга

**7. Пользовательский интерфейс:**
+ CLI (Command Line Interface) — командная строка (bash, PowerShell)
+ GUI (Graphical User Interface) — графический интерфейс

**Ядро (Kernel)** — центральная часть ОС, работающая в привилегированном режиме (kernel mode) с полным доступом к аппаратуре. Пользовательские приложения работают в user mode и обращаются к ядру через **системные вызовы (syscalls)**: `open()`, `read()`, `write()`, `fork()`, `exec()`, `mmap()` и т.д.

**Для Java-разработчика:**
+ JVM работает как обычный процесс ОС и использует все перечисленные механизмы
+ Java-потоки (`Thread`) отображаются на потоки ОС (в HotSpot — 1:1 mapping)
+ Java NIO использует системные вызовы (`epoll` в Linux, `IOCP` в Windows) для эффективного ввода/вывода

## Чем отличается процесс от потока?

**Процесс (Process)** — экземпляр выполняющейся программы. Процесс имеет собственное изолированное виртуальное адресное пространство, код, данные, открытые файлы, сетевые соединения и другие ресурсы.

**Поток (Thread)** — единица выполнения внутри процесса. Все потоки одного процесса разделяют одно адресное пространство и ресурсы процесса, но имеют собственный стек вызовов и набор регистров.

**Ключевые отличия:**

| Характеристика | Процесс | Поток |
|---|---|---|
| Адресное пространство | Собственное, изолированное | Общее с другими потоками процесса |
| Память | Своя куча (heap), стек, данные | Свой стек, но общая куча |
| Создание | Дорогое (~мс, копирование структур ОС) | Дешёвое (~мкс) |
| Переключение контекста | Дорогое (смена таблицы страниц, сброс TLB) | Дешевле (не нужно менять адресное пространство) |
| Обмен данными | Через IPC (pipes, shared memory, сокеты) | Напрямую через общую память |
| Изоляция | Полная (ошибка в одном процессе не затронет другой) | Низкая (ошибка в одном потоке может разрушить весь процесс) |
| Пример | Браузер Chrome — каждая вкладка в отдельном процессе | Веб-сервер — каждый запрос обрабатывается отдельным потоком |

**Контекст переключения (Context Switch):**

Когда ОС переключает выполнение с одного процесса/потока на другой, она:
1. Сохраняет состояние текущего (регистры процессора, Program Counter, указатель стека)
2. Загружает сохранённое состояние следующего
3. Для переключения процессов дополнительно: переключает таблицу страниц → сбрасывается TLB → следующие обращения к памяти вызывают TLB miss

Стоимость переключения:
+ Между потоками одного процесса: ~1–10 мкс
+ Между процессами: ~10–100 мкс (из-за сброса TLB)

**В Java:**
+ `java.lang.Thread` — создаёт поток ОС (в HotSpot JVM)
+ Виртуальные потоки (Virtual Threads, Java 21+) — лёгкие потоки, управляемые JVM, не привязанные 1:1 к потокам ОС. Тысячи виртуальных потоков могут работать на небольшом числе потоков ОС (carrier threads)
+ `ProcessBuilder` / `Runtime.exec()` — создание дочерних процессов из Java

## Что такое планировщик процессов (scheduler)?

**Планировщик процессов (scheduler)** — компонент ядра ОС, который определяет, какой процесс (или поток) будет выполняться на каком ядре процессора в каждый момент времени.

**Зачем нужен планировщик:**
В типичной системе количество запущенных потоков значительно превышает количество ядер CPU. Например, на 8-ядерном процессоре может быть запущено 200+ потоков. Планировщик создаёт иллюзию одновременного выполнения, быстро переключая контекст между потоками.

**Основные понятия:**

+ **Квант времени (time slice / quantum)** — максимальное время непрерывного выполнения процесса на CPU перед принудительным переключением. Обычно 1–10 мс
+ **Приоритет** — числовое значение, определяющее важность процесса. Процессы с более высоким приоритетом получают больше процессорного времени
+ **Вытесняющая многозадачность (preemptive multitasking)** — ОС может принудительно забрать CPU у процесса по истечении кванта или при появлении более приоритетного процесса. Используется во всех современных ОС

**Алгоритмы планирования:**

+ **Round Robin (RR)** — каждый процесс получает равный квант времени по очереди. Простой и справедливый
+ **Priority Scheduling** — процесс с наивысшим приоритетом выполняется первым. Проблема: голодание (starvation) — низкоприоритетные процессы могут никогда не получить CPU
+ **CFS (Completely Fair Scheduler)** — используется в Linux. Поддерживает «виртуальное время выполнения» для каждого процесса и выбирает тот, который выполнялся меньше всего. Реализован на основе красно-чёрного дерева
+ **Multilevel Feedback Queue** — несколько очередей с разными приоритетами. Процессы могут перемещаться между очередями в зависимости от поведения

**Типы процессов:**
+ **CPU-bound** — интенсивно используют процессор (вычисления, сжатие). Получают более длинные кванты
+ **I/O-bound** — часто ожидают ввода/вывода (сетевые запросы, чтение с диска). Получают более высокий приоритет, так как быстро освобождают CPU

**Для Java-разработчика:**
+ `Thread.setPriority()` — подсказка планировщику ОС, но не гарантия
+ Количество потоков в пуле (`ExecutorService`) следует выбирать с учётом характера задач: для CPU-bound ~ количество ядер, для I/O-bound — значительно больше
+ Виртуальные потоки (Java 21+) имеют собственный планировщик внутри JVM (ForkJoinPool), но carrier threads по-прежнему планируются ОС

## Что такое межпроцессное взаимодействие (IPC)?

**IPC (Inter-Process Communication)** — набор механизмов, позволяющих процессам обмениваться данными и синхронизировать свои действия. Поскольку процессы изолированы друг от друга (каждый имеет своё адресное пространство), для обмена данными им нужны специальные средства, предоставляемые ОС.

**Основные механизмы IPC:**

**1. Каналы (Pipes):**
+ **Неименованные (anonymous pipes)** — однонаправленный поток данных между связанными процессами (родитель-потомок). В shell: `ls | grep txt`
+ **Именованные (named pipes / FIFO)** — файл-канал в файловой системе, позволяющий общаться несвязанным процессам
+ Поток байтов (FIFO), без границ сообщений

**2. Разделяемая память (Shared Memory):**
+ Несколько процессов отображают один и тот же участок физической памяти в своё виртуальное адресное пространство
+ Самый быстрый способ IPC — нет копирования данных между процессами
+ Требует механизмов синхронизации (семафоры, мьютексы) для предотвращения состояния гонки

**3. Сокеты (Sockets):**
+ Универсальный механизм для обмена данными как между процессами на одной машине (Unix domain sockets), так и по сети (TCP/UDP sockets)
+ Двунаправленный обмен данными
+ Основной механизм сетевого взаимодействия

**4. Сигналы (Signals):**
+ Асинхронные уведомления, отправляемые процессу
+ Примеры: `SIGTERM` (запрос на завершение), `SIGKILL` (принудительное завершение), `SIGHUP` (перечитать конфигурацию), `SIGINT` (Ctrl+C)
+ Несут минимум информации (только номер сигнала)

**5. Очереди сообщений (Message Queues):**
+ Процессы отправляют и получают сообщения через именованную очередь
+ Сообщения сохраняются в очереди до тех пор, пока получатель их не прочитает
+ Поддерживают приоритеты сообщений

**6. Семафоры (Semaphores):**
+ Используются для синхронизации, а не для обмена данными
+ Счётчик, управляющий доступом к общему ресурсу
+ Бинарный семафор (0/1) работает как мьютекс

**7. Файлы и файловые блокировки:**
+ Процессы могут обмениваться данными через файл
+ Блокировки (`flock`) предотвращают одновременную запись

**Для Java-разработчика:**
+ `ProcessBuilder` — создание дочернего процесса и обмен данными через stdin/stdout (pipe)
+ `java.net.Socket` — сокеты TCP/UDP
+ `java.nio.channels.SocketChannel` — неблокирующий ввод/вывод
+ `MappedByteBuffer` (`FileChannel.map()`) — memory-mapped файлы, аналог shared memory
+ В продакшене для IPC между сервисами чаще используют: HTTP/REST, gRPC, очереди сообщений (Kafka, RabbitMQ)

## Что такое файловая система? Какие файловые системы вы знаете?

**Файловая система** — это способ организации, хранения и доступа к данным на накопителе. Файловая система определяет, как данные и метаданные (имя, размер, дата, права) размещаются на диске и как к ним осуществляется доступ.

**Основные понятия:**
+ **Файл** — именованная последовательность байтов
+ **Каталог (директория)** — специальный файл, содержащий список файлов и подкаталогов
+ **Путь** — адрес файла в иерархии каталогов (`/home/user/file.txt` или `C:\Users\file.txt`)
+ **Inode (index node)** — структура данных в Unix-подобных ФС, хранящая метаданные файла (размер, владелец, права, указатели на блоки данных). Имя файла хранится в каталоге, а не в inode

**Популярные файловые системы:**

| ФС | ОС | Макс. размер файла | Макс. размер тома | Особенности |
|---|---|---|---|---|
| **ext4** | Linux | 16 ТБ | 1 ЭБ | Стандартная ФС для Linux. Журналирование, extents, обратная совместимость с ext3 |
| **NTFS** | Windows | 16 ЭБ | 256 ТБ | Стандартная ФС для Windows. ACL, журналирование, шифрование (EFS), сжатие |
| **FAT32** | Кроссплатформенная | 4 ГБ | 2 ТБ | Простая, совместима со всеми ОС. Ограничение 4 ГБ на файл. Используется для USB-флешек |
| **exFAT** | Кроссплатформенная | 16 ЭБ | 128 ПБ | «FAT32 без ограничений». Для SD-карт и флешек |
| **XFS** | Linux | 8 ЭБ | 8 ЭБ | Высокая производительность для больших файлов. Используется в RHEL/CentOS |
| **ZFS** | Linux/FreeBSD | 16 ЭБ | 256 ЗБ | Проверка целостности данных (checksums), снапшоты, встроенный RAID |
| **APFS** | macOS | 8 ЭБ | 8 ЭБ | Оптимизирована для SSD, снапшоты, шифрование |

**Журналирование (Journaling):**
+ Перед изменением данных файловая система записывает намерение в журнал
+ Если произошёл сбой (отключение питания), ФС может восстановить целостность по журналу
+ ext4, NTFS, XFS — журналируемые ФС; FAT32 — нет

**Inode в ext4:**
+ Каждый файл и каталог имеет inode
+ Inode содержит: размер, владелец (UID/GID), права (rwxrwxrwx), временные метки (ctime, mtime, atime), указатели на блоки данных
+ Имя файла → номер inode (эта связь хранится в каталоге)
+ Один файл может иметь несколько имён (hard links), указывающих на один inode
+ Количество inode фиксировано при создании ФС

**Для Java-разработчика:**
+ `java.nio.file.Path`, `Files`, `FileSystem` — работа с файлами в Java (NIO.2)
+ `Files.getFileStore()` — информация о файловой системе
+ Ограничение FAT32 на 4 ГБ важно при записи больших файлов на USB

## Какие существуют уровни модели OSI?

**Модель OSI (Open Systems Interconnection)** — эталонная модель сетевого взаимодействия, разделяющая сетевую коммуникацию на 7 уровней. Каждый уровень выполняет определённые функции и взаимодействует только с соседними уровнями.

| # | Уровень | PDU (единица данных) | Функции | Примеры протоколов/технологий |
|--:|---------|:--------------------:|---------|:-----------------------------:|
| 7 | **Прикладной** (Application) | Данные | Интерфейс для пользовательских приложений, доступ к сетевым службам | HTTP, HTTPS, FTP, SMTP, DNS, SSH |
| 6 | **Представления** (Presentation) | Данные | Кодирование/декодирование данных, шифрование, сжатие | SSL/TLS, JPEG, JSON, XML |
| 5 | **Сеансовый** (Session) | Данные | Управление сеансами связи: установка, поддержание, завершение | RPC, NetBIOS, SIP |
| 4 | **Транспортный** (Transport) | Сегмент / Дейтаграмма | Надёжная доставка данных между хостами, контроль потока, мультиплексирование | TCP, UDP |
| 3 | **Сетевой** (Network) | Пакет | Логическая адресация, маршрутизация между сетями | IP, ICMP, ARP |
| 2 | **Канальный** (Data Link) | Кадр (Frame) | Физическая адресация, доступ к среде, обнаружение ошибок | Ethernet, Wi-Fi (802.11), PPP |
| 1 | **Физический** (Physical) | Биты | Передача битов через физическую среду | Витая пара, оптоволокно, радиоволны, USB |

**Принцип работы:**
+ **Отправка**: данные «спускаются» от уровня 7 к уровню 1. На каждом уровне к данным добавляется заголовок (инкапсуляция)
+ **Приём**: данные «поднимаются» от уровня 1 к уровню 7. На каждом уровне заголовок снимается (декапсуляция)

**Инкапсуляция на примере HTTP-запроса:**
```
[Ethernet Header][IP Header][TCP Header][HTTP Data][Ethernet Trailer]
     L2              L3          L4          L7            L2
```

**Мнемоника для запоминания (снизу вверх):** «Please Do Not Throw Sausage Pizza Away» (Physical, Data Link, Network, Transport, Session, Presentation, Application).

На практике модель OSI используется как теоретическая основа. Реальные сетевые протоколы ближе к модели TCP/IP, и границы между уровнями 5–7 часто размыты.

## Что такое модель TCP/IP и чем она отличается от OSI?

**Модель TCP/IP** — практическая модель сетевого взаимодействия, лежащая в основе Интернета. Состоит из 4 уровней.

| TCP/IP уровень | Соответствие OSI | Протоколы | Функции |
|---|---|---|---|
| **Прикладной** (Application) | 5 + 6 + 7 | HTTP, HTTPS, DNS, FTP, SMTP, SSH, DHCP | Протоколы приложений, представление данных, сеансы |
| **Транспортный** (Transport) | 4 | TCP, UDP | Доставка данных между приложениями, управление потоком |
| **Сетевой** (Internet) | 3 | IP, ICMP, ARP | Адресация и маршрутизация пакетов между сетями |
| **Канальный** (Network Access / Link) | 1 + 2 | Ethernet, Wi-Fi, PPP | Передача данных в рамках одной физической сети |

**Отличия TCP/IP от OSI:**

| Критерий | OSI | TCP/IP |
|---|---|---|
| Количество уровней | 7 | 4 |
| Происхождение | Теоретическая модель ISO | Практическая модель DARPA |
| Применение | Академическое описание | Реальная работа Интернета |
| Уровни 5–7 | Разделены | Объединены в «Прикладной» |
| Уровни 1–2 | Разделены | Объединены в «Канальный» |

На практике при обсуждении сетей чаще всего используют терминологию модели OSI (L2, L3, L4, L7), даже если реальная реализация ближе к TCP/IP.

## Что такое Ethernet, MAC-адрес, коммутатор и ARP?

Это понятия канального уровня (L2), отвечающего за передачу данных в пределах одной локальной сети.

**Ethernet:**
+ Технология и набор стандартов (IEEE 802.3) для проводных локальных сетей (LAN)
+ Определяет формат кадра (frame), правила доступа к среде передачи
+ Скорости: Fast Ethernet (100 Мбит/с), Gigabit Ethernet (1 Гбит/с), 10G Ethernet (10 Гбит/с)
+ Кадр Ethernet содержит: MAC-адрес получателя, MAC-адрес отправителя, тип протокола (EtherType), данные, контрольную сумму (FCS)

**MAC-адрес (Media Access Control Address):**
+ Уникальный идентификатор сетевого интерфейса, «прошитый» производителем
+ 48 бит (6 байт), записывается в шестнадцатеричном формате: `AA:BB:CC:DD:EE:FF`
+ Первые 3 байта — OUI (Organizationally Unique Identifier) — идентификатор производителя
+ Используется для адресации в пределах локальной сети (L2)
+ `FF:FF:FF:FF:FF:FF` — широковещательный (broadcast) адрес — кадр доставляется всем устройствам в сети

**Коммутатор (Switch):**
+ Устройство канального уровня, соединяющее устройства в локальной сети
+ Хранит **таблицу MAC-адресов** (CAM table): MAC-адрес → порт коммутатора
+ Получив кадр, смотрит MAC-адрес получателя в таблице и отправляет кадр только в нужный порт (в отличие от хаба, который отправляет всем)
+ Если MAC-адрес не найден — отправляет кадр во все порты (flooding)
+ Обучается автоматически: запоминает MAC-адрес отправителя и порт, с которого пришёл кадр

**ARP (Address Resolution Protocol):**
+ Протокол для определения MAC-адреса по известному IP-адресу
+ Работает в пределах одной локальной сети (L2-домена)

Процесс работы ARP:
1. Компьютер A хочет отправить пакет компьютеру B (IP: 192.168.1.5), но не знает его MAC-адрес
2. A отправляет **ARP-запрос** (broadcast на `FF:FF:FF:FF:FF:FF`): «У кого IP 192.168.1.5? Ответьте на мой MAC AA:BB:CC:11:22:33»
3. Все устройства в сети получают запрос, но отвечает только B
4. B отправляет **ARP-ответ** (unicast): «192.168.1.5 — это я, мой MAC: DD:EE:FF:44:55:66»
5. A сохраняет результат в **ARP-кэше** (временная таблица IP → MAC)

Посмотреть ARP-кэш: `arp -a` (Windows/Linux).

## Что такое IP-адресация? Чем отличаются IPv4 и IPv6?

**IP-адрес** — уникальный логический идентификатор устройства в сети на сетевом уровне (L3). Используется для маршрутизации пакетов между сетями.

**IPv4 (Internet Protocol version 4):**
+ 32-битный адрес, записывается как 4 десятичных числа через точку: `192.168.1.100`
+ Каждое число — 1 байт (0–255)
+ Всего 2³² ≈ 4.3 млрд адресов (что уже недостаточно)
+ Адрес делится на **сетевую часть** и **хостовую часть** (определяется маской подсети)

Специальные диапазоны IPv4:
+ `10.0.0.0/8` — частная сеть (класс A)
+ `172.16.0.0/12` — частная сеть (класс B)
+ `192.168.0.0/16` — частная сеть (класс C)
+ `127.0.0.0/8` — loopback (localhost), чаще всего `127.0.0.1`
+ `0.0.0.0` — все интерфейсы (при привязке сервера)
+ `255.255.255.255` — широковещательный адрес

**IPv6 (Internet Protocol version 6):**
+ 128-битный адрес, записывается как 8 групп по 4 hex-цифры через двоеточие: `2001:0db8:85a3:0000:0000:8a2e:0370:7334`
+ Можно сокращать: ведущие нули опускаются, последовательные группы нулей заменяются на `::` (один раз): `2001:db8:85a3::8a2e:370:7334`
+ 2¹²⁸ ≈ 3.4×10³⁸ адресов — хватит надолго
+ Нет NAT — каждое устройство может иметь глобально уникальный адрес
+ Нет broadcast — вместо него multicast
+ `::1` — loopback (аналог `127.0.0.1`)
+ `fe80::/10` — link-local адреса (аналог `169.254.x.x` в IPv4)

**Сравнение:**

| Свойство | IPv4 | IPv6 |
|---|---|---|
| Длина адреса | 32 бита | 128 бит |
| Формат | Десятичный (`192.168.1.1`) | Шестнадцатеричный (`2001:db8::1`) |
| Количество адресов | ~4.3 млрд | ~3.4×10³⁸ |
| Заголовок | Переменной длины | Фиксированной длины (40 байт), проще |
| NAT | Широко используется | Не нужен |
| IPSec | Опционально | Встроен |

**Для Java-разработчика:**
+ `InetAddress.getByName("192.168.1.1")` — работает и с IPv4, и с IPv6
+ При привязке серверного сокета на `0.0.0.0` — сервер слушает на всех IPv4-интерфейсах, на `::` — на всех (IPv4 + IPv6)
+ В URL IPv6 заключается в квадратные скобки: `http://[::1]:8080/`

## Что такое маска подсети и CIDR? Как разбивать сеть на подсети?

**Маска подсети (Subnet Mask)** — 32-битное число, определяющее, какая часть IP-адреса является адресом сети, а какая — адресом хоста. В двоичном виде маска — это последовательность единиц (сетевая часть), за которой следуют нули (хостовая часть).

**Примеры масок:**

| Маска | Двоичный вид | CIDR | Кол-во хостов |
|-------|-------------|------|---------------|
| `255.0.0.0` | `11111111.00000000.00000000.00000000` | /8 | 16 777 214 |
| `255.255.0.0` | `11111111.11111111.00000000.00000000` | /16 | 65 534 |
| `255.255.255.0` | `11111111.11111111.11111111.00000000` | /24 | 254 |
| `255.255.255.240` | `11111111.11111111.11111111.11110000` | /28 | 14 |

**CIDR (Classless Inter-Domain Routing):**
+ Компактная запись: IP-адрес / количество бит сетевой части
+ `192.168.1.0/24` означает: первые 24 бита — адрес сети, остальные 8 — хостовая часть
+ Заменил устаревшую классовую адресацию (класс A/B/C)

**Как вычислить адреса подсети:**

Дано: `192.168.1.100/26`

1. `/26` → маска: `255.255.255.192` (`11111111.11111111.11111111.11000000`)
2. Хостовых бит: 32 − 26 = 6. Количество адресов: 2⁶ = 64. Количество хостов: 64 − 2 = 62
3. **Адрес сети** (все хостовые биты = 0): `192.168.1.100 AND 255.255.255.192` = `192.168.1.64`
4. **Broadcast** (все хостовые биты = 1): `192.168.1.127`
5. Диапазон хостов: `192.168.1.65` — `192.168.1.126`

Пошагово для последнего октета:
```
100 в двоичном: 01100100
Маска (посл. октет): 11000000
AND:                  01000000 = 64 (адрес сети)
Broadcast:            01111111 = 127
```

**Разбиение сети на подсети:**

Если нужно разделить `10.0.0.0/24` (254 хоста) на 4 подсети:
+ Нужно дополнительно 2 бита для адресации 4 подсетей (2² = 4)
+ Новая маска: /24 + 2 = /26
+ Подсети:
  + `10.0.0.0/26` (хосты: .1 — .62)
  + `10.0.0.64/26` (хосты: .65 — .126)
  + `10.0.0.128/26` (хосты: .129 — .190)
  + `10.0.0.192/26` (хосты: .193 — .254)

**Для Java-разработчика:**
+ Понимание подсетей необходимо для настройки Docker-сетей, Kubernetes, firewall-правил, конфигурации серверов

## Чем отличаются TCP и UDP?

**TCP (Transmission Control Protocol)** и **UDP (User Datagram Protocol)** — два основных протокола транспортного уровня (L4).

| Свойство | TCP | UDP |
|---|---|---|
| Тип соединения | Ориентирован на соединение (connection-oriented) | Без установки соединения (connectionless) |
| Надёжность | Гарантированная доставка, повторная отправка при потере | Нет гарантии доставки |
| Порядок | Гарантирован (данные приходят в том же порядке) | Не гарантирован |
| Контроль потока | Есть (flow control, window) | Нет |
| Контроль перегрузки | Есть (congestion control) | Нет |
| Заголовок | 20–60 байт | 8 байт |
| Скорость | Ниже (из-за накладных расходов) | Выше |
| Тип данных | Поток байтов (stream) | Дейтаграммы (datagram, с границами сообщений) |

**TCP используется, когда:**
+ Нужна надёжная доставка без потерь: HTTP/HTTPS, SSH, FTP, SMTP, базы данных (JDBC)
+ Важен порядок данных
+ Можно позволить некоторую задержку

**UDP используется, когда:**
+ Важна скорость и минимальная задержка: онлайн-игры, голосовая/видеосвязь (VoIP, WebRTC), стриминг
+ Потеря отдельных пакетов допустима
+ DNS-запросы (обычно UDP, порт 53)
+ DHCP

**Механизмы надёжности TCP:**
+ **Sequence numbers** — каждый байт нумеруется, получатель собирает данные в правильном порядке
+ **ACK (Acknowledgment)** — получатель подтверждает получение данных
+ **Retransmission** — если ACK не получен за таймаут, данные отправляются повторно
+ **Checksum** — контрольная сумма для обнаружения повреждённых данных
+ **Flow control (Sliding Window)** — получатель сообщает, сколько данных готов принять, чтобы отправитель не перегрузил его
+ **Congestion control** — алгоритмы (Slow Start, Congestion Avoidance) для предотвращения перегрузки сети

## Что такое трёхстороннее рукопожатие (three-way handshake)?

**Three-way handshake** — процедура установления TCP-соединения между клиентом и сервером. Состоит из трёх шагов:

```
Клиент                          Сервер
   |                               |
   |------- SYN (seq=x) --------->|  1. Клиент хочет соединиться
   |                               |
   |<--- SYN-ACK (seq=y, ack=x+1)-|  2. Сервер подтверждает и предлагает свой номер
   |                               |
   |------- ACK (ack=y+1) ------->|  3. Клиент подтверждает
   |                               |
   |    Соединение установлено     |
```

**Шаг 1 — SYN (Synchronize):**
+ Клиент отправляет сегмент с флагом SYN и начальным номером последовательности (`seq=x`)
+ Клиент переходит в состояние `SYN_SENT`

**Шаг 2 — SYN-ACK:**
+ Сервер получает SYN, отвечает сегментом с флагами SYN и ACK
+ `seq=y` (свой начальный номер), `ack=x+1` (подтверждает получение SYN клиента)
+ Сервер переходит в состояние `SYN_RECEIVED`

**Шаг 3 — ACK:**
+ Клиент отправляет ACK с `ack=y+1` (подтверждает получение SYN сервера)
+ Оба переходят в состояние `ESTABLISHED` — соединение установлено, можно передавать данные

**Зачем три шага, а не два?**
+ Оба участника должны согласовать начальные номера последовательностей (ISN — Initial Sequence Number) для каждого направления
+ Два шага не гарантируют, что обе стороны готовы к передаче данных
+ Защита от дублированных SYN-пакетов от прошлых соединений

**Завершение TCP-соединения (four-way handshake):**
```
Клиент                          Сервер
   |------- FIN ----------------->|  1. Клиент хочет завершить
   |<------- ACK -----------------|  2. Сервер подтверждает
   |<------- FIN -----------------|  3. Сервер тоже завершает
   |------- ACK ----------------->|  4. Клиент подтверждает
```

После закрытия клиент переходит в состояние `TIME_WAIT` (обычно 2×MSL ≈ 60 секунд), чтобы гарантировать, что последний ACK дошёл до сервера.

**Для Java-разработчика:**
+ `new Socket("host", port)` — внутри происходит three-way handshake
+ `socket.close()` — four-way handshake
+ При большом количестве короткоживущих соединений состояния `TIME_WAIT` могут исчерпать ресурсы. Решение: HTTP keep-alive, connection pooling (HikariCP для JDBC)

## Что такое порты? Какие well-known порты вы знаете?

**Порт** — 16-битное число (0–65535), идентифицирующее конкретное приложение (или сервис) на хосте. Комбинация `IP-адрес:порт` однозначно определяет конечную точку (endpoint) сетевого взаимодействия.

Порт позволяет нескольким приложениям на одном компьютере одновременно использовать сеть. Когда пакет приходит на хост, ОС смотрит на номер порта назначения и передаёт данные соответствующему процессу.

**Диапазоны портов:**

| Диапазон | Название | Описание |
|----------|---------|----------|
| 0–1023 | **Well-known (системные)** | Зарезервированы для стандартных сервисов. На Linux требуют root-прав для привязки |
| 1024–49151 | **Registered (зарегистрированные)** | Зарегистрированы IANA для конкретных приложений |
| 49152–65535 | **Ephemeral (динамические)** | Назначаются ОС автоматически для исходящих соединений клиентов |

**Основные well-known порты:**

| Порт | Протокол | Сервис |
|------|----------|--------|
| 20, 21 | TCP | FTP (данные / управление) |
| 22 | TCP | SSH (удалённый доступ) |
| 23 | TCP | Telnet |
| 25 | TCP | SMTP (отправка email) |
| 53 | TCP/UDP | DNS |
| 80 | TCP | HTTP |
| 110 | TCP | POP3 (получение email) |
| 143 | TCP | IMAP (получение email) |
| 443 | TCP | HTTPS (HTTP + TLS) |
| 3306 | TCP | MySQL |
| 5432 | TCP | PostgreSQL |
| 6379 | TCP | Redis |
| 8080 | TCP | HTTP (альтернативный порт, Tomcat по умолчанию) |
| 8443 | TCP | HTTPS (альтернативный порт) |
| 9092 | TCP | Apache Kafka |
| 27017 | TCP | MongoDB |

**Для Java-разработчика:**
+ Spring Boot по умолчанию запускается на порту `8080` (настройка `server.port`)
+ На одном порту может слушать только один процесс. `java.net.BindException: Address already in use` — порт уже занят
+ `netstat -tlnp` (Linux) или `netstat -ano` (Windows) — посмотреть, какие порты заняты
+ В Docker при `-p 8080:8080` первый порт — хостовой, второй — контейнерный

## Как работает DNS?

**DNS (Domain Name System)** — распределённая иерархическая система, преобразующая доменные имена (например, `www.google.com`) в IP-адреса (например, `142.250.74.68`). DNS — это «телефонная книга» Интернета.

**Иерархия DNS:**

```
                .  (корневой домен)
               / \
           .com   .ru   .org   ...
            |       |
        google    yandex
          |         |
        www       mail
```

+ **Корневые серверы** — 13 групп серверов (от a.root-servers.net до m.root-servers.net), знающие адреса серверов доменов верхнего уровня
+ **TLD (Top-Level Domain)** — домены верхнего уровня: `.com`, `.ru`, `.org`, `.net`
+ **Авторитативные серверы** — хранят DNS-записи для конкретных доменов

**Процесс разрешения DNS (рекурсивный запрос):**

Пример: браузер запрашивает `www.example.com`

1. **Кэш браузера** — проверяется локальный кэш. Если запись есть и не устарела (TTL не истёк) — возвращается сразу
2. **Кэш ОС** — проверяется системный кэш DNS + файл `hosts` (`/etc/hosts` или `C:\Windows\System32\drivers\etc\hosts`)
3. **Рекурсивный DNS-резолвер** — обычно DNS-сервер провайдера или публичный (8.8.8.8, 1.1.1.1). Если ответ в его кэше — возвращается клиенту
4. Если нет в кэше, резолвер начинает **итеративный запрос**:
   + Спрашивает у **корневого сервера**: «Где `.com`?» → получает адрес TLD-сервера для `.com`
   + Спрашивает у **TLD-сервера `.com`**: «Где `example.com`?» → получает адрес авторитативного сервера для `example.com`
   + Спрашивает у **авторитативного сервера**: «Какой IP у `www.example.com`?» → получает ответ: `93.184.216.34`
5. Резолвер кэширует результат (на время TTL) и возвращает его клиенту
6. Браузер устанавливает TCP-соединение с полученным IP

**TTL (Time To Live)** — время в секундах, в течение которого DNS-запись может кэшироваться. Низкий TTL (60 с) позволяет быстро менять IP, но увеличивает нагрузку на DNS. Высокий TTL (86400 с = 24 ч) снижает нагрузку, но изменения распространяются медленно.

**Для Java-разработчика:**
+ JVM кэширует DNS-ответы (по умолчанию навсегда для положительных ответов в SecurityManager, 30 с без него). Это может быть проблемой при использовании load balancer с меняющимися IP
+ Настройка: `networkaddress.cache.ttl` в `java.security`
+ `InetAddress.getByName("example.com")` — резолвинг DNS в Java

## Какие типы DNS-записей существуют?

DNS-записи (Resource Records) хранятся на авторитативных DNS-серверах и определяют различные свойства домена.

**Основные типы записей:**

| Тип | Назначение | Пример |
|-----|-----------|--------|
| **A** | Сопоставляет доменное имя с IPv4-адресом | `example.com. A 93.184.216.34` |
| **AAAA** | Сопоставляет доменное имя с IPv6-адресом | `example.com. AAAA 2606:2800:220:1:248:1893:25c8:1946` |
| **CNAME** | Каноническое имя (alias). Указывает, что домен является псевдонимом другого домена | `www.example.com. CNAME example.com.` |
| **MX** | Mail Exchange. Указывает сервер для приёма email для домена. Имеет приоритет (чем ниже число, тем выше приоритет) | `example.com. MX 10 mail.example.com.` |
| **NS** | Name Server. Указывает авторитативные DNS-серверы для домена | `example.com. NS ns1.example.com.` |
| **TXT** | Произвольная текстовая информация. Используется для SPF (защита от подделки email), DKIM, верификации домена | `example.com. TXT "v=spf1 include:_spf.google.com ~all"` |
| **SOA** | Start of Authority. Метаинформация о зоне: основной NS, email администратора, серийный номер, интервалы обновления | Одна запись на зону |
| **PTR** | Pointer. Обратная запись: IP → доменное имя (reverse DNS) | `34.216.184.93.in-addr.arpa. PTR example.com.` |
| **SRV** | Service. Указывает хост и порт для определённого сервиса | `_sip._tcp.example.com. SRV 10 60 5060 sip.example.com.` |

**Часто используемые комбинации:**
+ **A + CNAME**: `example.com → A → 1.2.3.4`, `www.example.com → CNAME → example.com`
+ **MX**: несколько записей с разными приоритетами для резервирования почтовых серверов
+ **TXT для SPF/DKIM/DMARC**: защита от спама и фишинга

**Утилиты для проверки DNS:**
+ `nslookup example.com` — простой запрос
+ `dig example.com A` — подробный запрос (Linux)
+ `dig +trace example.com` — показывает весь путь рекурсивного разрешения

## Как работает HTTP/HTTPS? Что такое TLS?

**HTTP (HyperText Transfer Protocol)** — протокол прикладного уровня для передачи данных в Интернете. Основан на модели «запрос-ответ» (request-response): клиент отправляет запрос, сервер возвращает ответ.

**Структура HTTP-запроса:**
```
GET /api/users HTTP/1.1        ← Метод, путь, версия
Host: example.com              ← Заголовки
Content-Type: application/json
Authorization: Bearer token123
                               ← Пустая строка
{"name": "John"}               ← Тело (body, для POST/PUT)
```

**Структура HTTP-ответа:**
```
HTTP/1.1 200 OK                ← Версия, код статуса, описание
Content-Type: application/json ← Заголовки
Content-Length: 27
                               ← Пустая строка
{"id": 1, "name": "John"}     ← Тело ответа
```

**HTTP — незашифрованный протокол.** Данные передаются открытым текстом, что позволяет перехватить их (man-in-the-middle).

**HTTPS (HTTP Secure):**
+ HTTP, обёрнутый в шифрование TLS (ранее SSL)
+ Обеспечивает: **конфиденциальность** (шифрование), **целостность** (данные не изменены), **аутентификацию** (сервер тот, за кого себя выдаёт)
+ По умолчанию порт 443

**TLS (Transport Layer Security):**

TLS Handshake (упрощённо, TLS 1.2):
```
Клиент                                Сервер
   |--- ClientHello ------------------>|  Версия TLS, поддерживаемые шифры, random
   |<-- ServerHello -------------------|  Выбранный шифр, random
   |<-- Certificate -------------------|  Сертификат сервера (X.509)
   |<-- ServerHelloDone ---------------|
   |--- ClientKeyExchange ------------>|  Pre-master secret (зашифрован публичным ключом сервера)
   |--- ChangeCipherSpec ------------->|  Переключаюсь на шифрование
   |--- Finished --------------------->|  Зашифрованное подтверждение
   |<-- ChangeCipherSpec --------------|  Сервер тоже переключается
   |<-- Finished ----------------------|  Зашифрованное подтверждение
   |                                   |
   |=== Зашифрованный HTTP-трафик ====|
```

В TLS 1.3 handshake упрощён до 1 RTT (round-trip), что быстрее.

**Сертификат X.509:**
+ Содержит: доменное имя, публичный ключ сервера, срок действия, имя издателя (CA)
+ Подписан **удостоверяющим центром (CA — Certificate Authority)**: Let's Encrypt, DigiCert, Comodo
+ Клиент (браузер) проверяет подпись сертификата по цепочке доверия до корневого CA, встроенного в ОС/браузер

**Для Java-разработчика:**
+ Spring Boot HTTPS: настройка через `server.ssl.*` в `application.properties`
+ Java хранит доверенные CA-сертификаты в **truststore** (`$JAVA_HOME/lib/security/cacerts`)
+ `RestTemplate`, `WebClient`, `HttpClient` — поддерживают HTTPS из коробки

## Чем отличаются HTTP/1.1, HTTP/2 и HTTP/3?

| Свойство | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|
| Год | 1997 | 2015 | 2022 |
| Транспорт | TCP | TCP | QUIC (поверх UDP) |
| Формат | Текстовый | Бинарный | Бинарный |
| Мультиплексирование | Нет (один запрос за раз на соединение) | Да (множество потоков в одном TCP-соединении) | Да (потоки независимы) |
| Head-of-line blocking | На уровне HTTP и TCP | Только на уровне TCP | Нет |
| Сжатие заголовков | Нет | HPACK | QPACK |
| Server Push | Нет | Да | Да (редко используется) |
| Шифрование | Опционально (HTTPS) | Фактически обязательно | Обязательно (встроено в QUIC) |

**HTTP/1.1:**
+ Один запрос за раз на одно TCP-соединение (pipelining существует, но на практике не работает из-за head-of-line blocking)
+ Для параллельности браузеры открывают 6–8 TCP-соединений к одному серверу
+ Заголовки передаются текстом, повторяются в каждом запросе (Cookie, User-Agent — много байт)
+ Оптимизации: keep-alive (переиспользование соединений), конкатенация файлов, спрайты

**HTTP/2:**
+ **Бинарный протокол** — данные передаются в виде фреймов (frames), что эффективнее парсинга текста
+ **Мультиплексирование** — множество запросов и ответов передаются одновременно по одному TCP-соединению, чередуясь фреймами. Не нужно ждать завершения предыдущего запроса
+ **Сжатие заголовков (HPACK)** — заголовки сжимаются и кэшируются; повторяющиеся заголовки передаются как ссылки
+ **Server Push** — сервер может отправить ресурсы до того, как клиент их запросит (например, CSS к HTML-странице)
+ **Проблема:** head-of-line blocking на уровне TCP — если один TCP-пакет потерян, все потоки HTTP/2 ждут его повторной передачи

**HTTP/3:**
+ Использует **QUIC** — транспортный протокол поверх UDP, разработанный Google
+ Каждый HTTP-поток в QUIC независим: потеря пакета одного потока не блокирует другие
+ Встроенное шифрование (TLS 1.3) — handshake QUIC и TLS происходят вместе, что быстрее
+ **0-RTT** — при повторном подключении к серверу данные могут отправляться сразу, без ожидания handshake
+ Лучше работает при нестабильном соединении (мобильные сети)

**Для Java-разработчика:**
+ `java.net.http.HttpClient` (Java 11+) поддерживает HTTP/1.1 и HTTP/2
+ Spring Boot (Tomcat, Jetty) поддерживает HTTP/2 с минимальной конфигурацией
+ Для HTTP/3 пока нужны сторонние библиотеки или reverse proxy (Nginx, Caddy)

## Что такое WebSocket и чем он отличается от HTTP?

**WebSocket** — протокол, обеспечивающий постоянное двунаправленное (full-duplex) соединение между клиентом и сервером поверх TCP. Определён в RFC 6455.

**Отличия от HTTP:**

| Свойство | HTTP | WebSocket |
|---|---|---|
| Модель | Запрос-ответ (request-response) | Двунаправленный обмен (full-duplex) |
| Инициатор | Только клиент отправляет запрос | Обе стороны могут отправлять сообщения в любой момент |
| Соединение | Создаётся на каждый запрос (или keep-alive) | Постоянное, пока явно не закрыто |
| Заголовки | В каждом запросе/ответе | Только при установлении соединения (handshake) |
| Протокол | `http://` / `https://` | `ws://` / `wss://` (шифрованный) |
| Накладные расходы | Высокие (заголовки в каждом сообщении) | Низкие (фрейм 2–14 байт) |

**Установление WebSocket-соединения (handshake):**

WebSocket начинается как обычный HTTP-запрос с upgrade:

```
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
```

Сервер отвечает:
```
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

После этого соединение переключается с HTTP на WebSocket, и обе стороны обмениваются фреймами напрямую.

**Когда использовать WebSocket:**
+ Чаты, мессенджеры
+ Уведомления в реальном времени (notifications, alerts)
+ Онлайн-игры
+ Биржевые котировки, live-данные
+ Collaborative editing (совместное редактирование)

**Когда НЕ использовать WebSocket:**
+ Обычные CRUD-операции (REST/HTTP достаточно)
+ Нечастые запросы — long polling или SSE (Server-Sent Events) проще

**Для Java-разработчика:**
+ Spring WebSocket (`@EnableWebSocket`, `WebSocketHandler`)
+ Spring STOMP over WebSocket (`@MessageMapping`) — более высокоуровневый подход с pub/sub
+ Jakarta WebSocket API (`@ServerEndpoint`)

## Что такое NAT и зачем он нужен?

**NAT (Network Address Translation)** — механизм преобразования IP-адресов в заголовках пакетов при их прохождении через маршрутизатор. Обычно NAT заменяет приватный (локальный) IP-адрес на публичный (внешний) и обратно.

**Зачем нужен NAT:**
+ **Экономия IPv4-адресов** — организация может иметь один публичный IP на маршрутизаторе, а внутри сети использовать приватные адреса (192.168.x.x) для сотен устройств
+ **Безопасность** — устройства за NAT не видны из Интернета напрямую, что затрудняет атаки
+ **Гибкость** — можно менять внутреннюю структуру сети, не меняя внешний IP

**Как работает NAT (PAT / NAPT — Port Address Translation):**

```
Внутренняя сеть              Маршрутизатор (NAT)              Интернет
                             Публичный IP: 203.0.113.5

PC1 (192.168.1.10:5000) --→ NAT → 203.0.113.5:40001 -----→ Сервер (93.184.216.34:80)
PC2 (192.168.1.20:5000) --→ NAT → 203.0.113.5:40002 -----→ Сервер (93.184.216.34:80)
```

1. PC1 отправляет пакет: src=192.168.1.10:5000, dst=93.184.216.34:80
2. NAT заменяет src на 203.0.113.5:40001 и записывает соответствие в **NAT-таблицу**
3. Сервер отвечает на 203.0.113.5:40001
4. NAT смотрит в таблицу, заменяет dst обратно на 192.168.1.10:5000 и пересылает PC1

**Типы NAT:**
+ **SNAT (Source NAT)** — подмена адреса источника. Самый распространённый: внутренние устройства выходят в Интернет через один публичный IP
+ **DNAT (Destination NAT)** — подмена адреса назначения. Используется для перенаправления входящих соединений на внутренний сервер (port forwarding)
+ **PAT (Port Address Translation) / NAPT** — NAT с трансляцией портов. Множество устройств используют один IP, различаясь портами

**Проблемы NAT:**
+ Некоторые протоколы (FTP, SIP, P2P) плохо работают через NAT
+ Невозможно подключиться извне к устройству за NAT без port forwarding
+ Усложняет P2P-соединения (нужен STUN/TURN для WebRTC)
+ При IPv6 NAT не нужен — адресов достаточно для каждого устройства

**Для Java-разработчика:**
+ При развёртывании приложения важно понимать: приложение внутри Docker-контейнера находится за NAT (Docker network). Порт пробрасывается через `-p`
+ Port forwarding необходим, чтобы внешний трафик достигал серверного приложения за NAT

## Как работает маршрутизация? Что такое таблица маршрутизации и default gateway?

**Маршрутизация (Routing)** — процесс определения пути, по которому пакет должен быть отправлен от источника к назначению через сеть. Выполняется на сетевом уровне (L3) маршрутизаторами (и хостами).

**Таблица маршрутизации (Routing Table):**

Каждый маршрутизатор (и каждый компьютер) имеет таблицу маршрутизации — набор правил, определяющих, куда отправлять пакеты.

Пример таблицы маршрутизации:

| Сеть назначения | Маска | Шлюз (Gateway) | Интерфейс | Метрика |
|---|---|---|---|---|
| 192.168.1.0 | /24 | * (напрямую) | eth0 | 0 |
| 10.0.0.0 | /8 | 192.168.1.1 | eth0 | 10 |
| 0.0.0.0 | /0 | 192.168.1.1 | eth0 | 100 |

Процесс принятия решения:
1. Маршрутизатор получает пакет и смотрит на **IP-адрес назначения**
2. Сравнивает адрес с каждой записью в таблице (с помощью побитового AND с маской)
3. Выбирает наиболее конкретный маршрут (**longest prefix match** — маршрут с самой длинной маской)
4. Если ни один маршрут не подходит — используется **default route** (0.0.0.0/0)

**Default Gateway (шлюз по умолчанию):**
+ Маршрутизатор, которому хост отправляет пакеты, если адрес назначения не принадлежит локальной сети
+ Обычно это IP-адрес маршрутизатора (роутера) в локальной сети, например `192.168.1.1`
+ Маршрут по умолчанию: `0.0.0.0/0 via 192.168.1.1` — «всё, что не знаю куда отправить, отправляю на 192.168.1.1»

**Статическая vs динамическая маршрутизация:**
+ **Статическая** — маршруты прописываются вручную администратором. Подходит для небольших сетей
+ **Динамическая** — маршрутизаторы автоматически обмениваются информацией о маршрутах с помощью протоколов:
  + **OSPF (Open Shortest Path First)** — для внутренней маршрутизации (IGP)
  + **BGP (Border Gateway Protocol)** — для маршрутизации между автономными системами (Интернет-провайдерами). Основа маршрутизации в Интернете

**Утилиты:**
+ `route print` (Windows) / `ip route` (Linux) — просмотр таблицы маршрутизации
+ `traceroute` (Linux) / `tracert` (Windows) — отображение пути пакета через маршрутизаторы

## Как пакет проходит путь от клиента до сервера через сеть?

Рассмотрим полный путь HTTP-запроса от браузера к серверу `www.example.com`:

**1. Разрешение DNS:**
+ Браузер проверяет кэш → ОС проверяет кэш и `hosts` → DNS-резолвер
+ Получаем IP: `93.184.216.34`

**2. Формирование HTTP-запроса (L7):**
```
GET /index.html HTTP/1.1
Host: www.example.com
```

**3. Установление TCP-соединения (L4):**
+ ОС выбирает эфемерный порт (например, 52431)
+ Three-way handshake: SYN → SYN-ACK → ACK
+ Если HTTPS — TLS handshake

**4. Сегментация (L4):**
+ TCP разбивает данные на сегменты (обычно до MSS ~1460 байт)
+ Каждый сегмент получает заголовок TCP (src port: 52431, dst port: 443, seq number и т.д.)

**5. Добавление IP-заголовка (L3):**
+ К каждому сегменту добавляется IP-заголовок: src IP: 192.168.1.10, dst IP: 93.184.216.34, TTL, protocol=TCP
+ ОС проверяет таблицу маршрутизации: `93.184.216.34` не в локальной сети → отправить на default gateway `192.168.1.1`

**6. Определение MAC-адреса (L2):**
+ ОС знает IP шлюза: `192.168.1.1`. Нужен его MAC-адрес
+ Проверяет ARP-кэш. Если нет — ARP-запрос (broadcast) → ARP-ответ
+ Получает MAC: `AA:BB:CC:DD:EE:01`

**7. Формирование Ethernet-кадра (L2):**
+ Добавляется Ethernet-заголовок: src MAC (карты клиента), dst MAC (маршрутизатора), EtherType (IPv4)
+ Добавляется FCS (контрольная сумма)

**8. Передача по физической среде (L1):**
+ Кадр преобразуется в электрические сигналы (витая пара) / световые импульсы (оптоволокно) / радиоволны (Wi-Fi)

**9. На маршрутизаторе (default gateway):**
+ Получает кадр, проверяет FCS, снимает L2-заголовок
+ Смотрит IP назначения в таблице маршрутизации → определяет следующий hop
+ NAT: заменяет src IP на публичный, записывает в NAT-таблицу
+ Создаёт новый L2-кадр с MAC следующего маршрутизатора
+ Отправляет кадр дальше

**10. Через Интернет:**
+ Пакет проходит через несколько маршрутизаторов (hops), каждый из которых повторяет шаг 9
+ На каждом хопе L2-заголовок меняется (MAC-адреса), но L3 (IP-адреса) остаётся (кроме NAT)

**11. На сервере:**
+ Пакет приходит, L2 и L3 заголовки снимаются
+ TCP-стек собирает сегменты в поток данных, отправляет ACK
+ Данные передаются приложению (веб-серверу)

**12. Ответ проходит обратный путь.**

## Что такое VPN? Какие типы VPN существуют?

**VPN (Virtual Private Network)** — технология, создающая зашифрованный туннель между устройствами через публичную сеть (Интернет). VPN позволяет передавать данные так, будто устройства находятся в одной локальной сети, даже если они физически разнесены.

**Зачем нужен VPN:**
+ **Безопасность** — шифрование трафика при использовании публичного Wi-Fi
+ **Удалённый доступ** — подключение к корпоративной сети из дома
+ **Объединение офисов** — Site-to-Site VPN между филиалами
+ **Обход ограничений** — доступ к ресурсам, ограниченным по географии
+ **Конфиденциальность** — скрытие реального IP-адреса

**Принцип работы (туннелирование):**

```
[Исходный пакет: IP-заголовок + Данные]
                ↓ шифрование и инкапсуляция
[Внешний IP-заголовок][VPN-заголовок][Зашифрованный исходный пакет]
```

Исходный пакет полностью шифруется и помещается внутрь нового пакета с новыми IP-заголовками. Промежуточные маршрутизаторы видят только внешний IP и не могут прочитать содержимое.

**Типы VPN:**

**IPSec (Internet Protocol Security):**
+ Работает на сетевом уровне (L3)
+ Два режима: транспортный (шифрует только данные) и туннельный (шифрует весь пакет)
+ Протоколы: ESP (шифрование + аутентификация), AH (только аутентификация)
+ IKE (Internet Key Exchange) для согласования ключей
+ Широко используется для Site-to-Site VPN и в корпоративных сетях

**OpenVPN:**
+ Open-source решение
+ Работает поверх TCP или UDP (обычно UDP порт 1194)
+ Использует TLS для обмена ключами, OpenSSL для шифрования
+ Гибкая конфигурация, кроссплатформенность
+ Хорошо проходит через NAT и firewall

**WireGuard:**
+ Современный, минималистичный протокол (~4000 строк кода)
+ Работает поверх UDP
+ Использует современную криптографию (Curve25519, ChaCha20, Poly1305)
+ Высокая производительность и низкая задержка
+ Встроен в ядро Linux (с версии 5.6)
+ Проще в настройке, чем IPSec и OpenVPN

**Для Java-разработчика:**
+ При работе через VPN приложение подключается к ресурсам (БД, микросервисы) по их внутренним (приватным) IP-адресам
+ VPN может влиять на MTU (Maximum Transmission Unit) — из-за дополнительных заголовков пакеты могут фрагментироваться
+ В Kubernetes часто используются сетевые решения (CNI-плагины), работающие по принципу VPN/overlay-сети

## Что такое Proxy и Reverse Proxy?

**Proxy (прямой прокси, forward proxy)** — сервер-посредник между клиентом и целевым сервером. Клиент явно отправляет запросы через прокси, и прокси обращается к серверу от имени клиента.

```
Клиент → [Forward Proxy] → Интернет → Сервер
```

**Зачем нужен Forward Proxy:**
+ **Анонимность** — скрывает IP клиента от сервера
+ **Кэширование** — прокси хранит кэш ответов, ускоряя повторные запросы
+ **Фильтрация** — корпоративный прокси блокирует доступ к определённым сайтам
+ **Обход ограничений** — доступ к ресурсам через прокси в другой стране

**Reverse Proxy (обратный прокси)** — сервер-посредник, расположенный перед серверами. Клиент обращается к reverse proxy, не зная о реальных серверах за ним.

```
Клиент → [Reverse Proxy (Nginx)] → Backend Server 1
                                  → Backend Server 2
                                  → Backend Server 3
```

**Зачем нужен Reverse Proxy:**
+ **Балансировка нагрузки (Load Balancing)** — распределяет запросы между несколькими backend-серверами (round-robin, least connections, IP hash)
+ **SSL/TLS-терминация** — reverse proxy обрабатывает HTTPS (TLS handshake), а к backend обращается по HTTP. Снимает криптографическую нагрузку с backend
+ **Кэширование** — кэширует статические ресурсы (изображения, CSS, JS)
+ **Сжатие** — gzip/brotli-сжатие ответов
+ **Защита** — скрывает структуру backend, может фильтровать вредоносные запросы, ограничивать rate limit
+ **Обслуживание статики** — раздаёт статические файлы напрямую, не нагружая backend

**Nginx как Reverse Proxy:**

```nginx
upstream backend {
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
    server 10.0.0.3:8080;
}

server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    location /api/ {
        proxy_pass http://backend;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    location / {
        root /var/www/static;
    }
}
```

**Для Java-разработчика:**
+ Типичная production-схема: **Nginx** (reverse proxy, SSL, статика) → **Spring Boot** (API, бизнес-логика) → **PostgreSQL**
+ Заголовки `X-Forwarded-For`, `X-Real-IP` — передают реальный IP клиента через прокси. В Spring: `server.forward-headers-strategy=native`
+ В Docker/Kubernetes Ingress Controller (Nginx Ingress) выполняет роль reverse proxy

## Что такое CDN и зачем он нужен?

**CDN (Content Delivery Network)** — географически распределённая сеть серверов, предназначенная для быстрой доставки контента пользователям. CDN размещает копии контента на серверах (edge servers / PoP — Point of Presence) по всему миру, чтобы пользователь получал данные с ближайшего к нему сервера.

**Как работает CDN:**

```
Без CDN:
Пользователь (Москва) ────── 200 мс ──────→ Origin-сервер (Нью-Йорк)

С CDN:
Пользователь (Москва) ── 20 мс ──→ CDN Edge (Москва) ── кэш hit → ответ
                                                        ── кэш miss → Origin → кэширование → ответ
```

1. Пользователь запрашивает `https://cdn.example.com/image.jpg`
2. DNS направляет запрос на ближайший edge-сервер CDN (через anycast или geo-DNS)
3. Если контент есть в кэше edge-сервера (cache hit) — возвращается сразу (~20 мс)
4. Если нет (cache miss) — edge-сервер запрашивает origin-сервер, кэширует ответ и возвращает пользователю

**Зачем нужен CDN:**
+ **Снижение задержки (latency)** — контент ближе к пользователю → быстрее загрузка
+ **Снижение нагрузки на origin** — большинство запросов обслуживается кэшем CDN
+ **Высокая доступность** — если один edge-сервер недоступен, запрос перенаправляется на другой
+ **Защита от DDoS** — CDN поглощает трафик атаки на своих серверах
+ **Пропускная способность** — CDN может обслуживать огромные объёмы трафика (видео, обновления ПО)

**Что обычно раздаётся через CDN:**
+ Статические файлы: изображения, CSS, JavaScript, шрифты
+ Видеостриминг (Netflix, YouTube)
+ Загрузка файлов (обновления ОС, дистрибутивы)
+ API-ответы (с правильными заголовками кэширования)

**Популярные CDN-провайдеры:**
+ Cloudflare
+ AWS CloudFront
+ Akamai
+ Google Cloud CDN
+ Fastly

**Заголовки кэширования (важны для CDN):**
+ `Cache-Control: public, max-age=31536000` — можно кэшировать публично, 1 год
+ `Cache-Control: no-cache` — нужно проверять у origin перед отдачей
+ `Cache-Control: private` — кэшировать только у клиента, не на CDN
+ `ETag` / `Last-Modified` — для условных запросов (304 Not Modified)

**Для Java-разработчика:**
+ Статические ресурсы приложения (JS, CSS) лучше отдавать через CDN
+ Правильная настройка `Cache-Control` в Spring Boot для статических ресурсов:
```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        registry.addResourceHandler("/static/**")
                .addResourceLocations("classpath:/static/")
                .setCacheControl(CacheControl.maxAge(365, TimeUnit.DAYS));
    }
}
```
+ API-ответы обычно не кэшируются на CDN (если не настроено явно), но иногда это полезно для read-heavy API
