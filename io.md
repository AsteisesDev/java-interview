## В чём заключается разница между IO и NIO?
+ Java IO (input-output) является потокоориентированным, а Java NIO (new/non-blocking io) – буфер-ориентированным. Потокоориентированный ввод/вывод подразумевает чтение/запись из потока/в поток одного или нескольких байт в единицу времени поочередно. Данная информация нигде не кэшируются. Таким образом, невозможно произвольно двигаться по потоку данных вперед или назад. В Java NIO данные сначала считываются в буфер, что дает больше гибкости при обработке данных.
+ Потоки ввода/вывода в Java IO являются блокирующими. Это значит, что когда в потоке выполнения вызывается `read()` или `write()` метод любого класса из пакета `java.io.*`, происходит блокировка до тех пор, пока данные не будут считаны или записаны. Поток выполнения в данный момент не может делать ничего другого. Неблокирующий режим Java NIO позволяет запрашивать считанные данные из канала (channel) и получать только то, что доступно на данный момент, или вообще ничего, если доступных данных пока нет. Вместо того, чтобы оставаться заблокированным пока данные не станут доступными для считывания, поток выполнения может заняться чем-то другим. Тоже самое справедливо и для неблокирующего вывода. Поток выполнения может запросить запись в канал некоторых данных, но не дожидаться при этом пока они не будут полностью записаны.
+ В Java NIO имеются селекторы, которые позволяют одному потоку выполнения мониторить несколько каналов ввода. Т.е. существует возможность зарегистрировать несколько каналов с селектором, а потом использовать один поток выполнения для обслуживания каналов, имеющих доступные для обработки данные, или для выбора каналов, готовых для записи.

## Какие особенности NIO вы знаете?
+ __Каналы и селекторы__: NIO поддерживает различные типы каналов. Канал является абстракцией объектов более низкого уровня файловой системы (например, отображенные в памяти файлы и блокировки файлов), что позволяет передавать данные с более высокой скоростью. Каналы не блокируются и поэтому Java предоставляет еще такие инструменты, как селектор, который позволяет выбрать готовый канал для передачи данных, и сокет, который является инструментом для блокировки.
+ __Буферы__: имеет буферизация для всех классов-обёрток примитивов (кроме Boolean). Появился абстрактный класс Buffer, который предоставляет такие операции, как clear, flip, mark и т.д. Его подклассы предоставляют методы для получения и установки данных.
+ __Кодировки__: появились кодеры и декодеры для отображения байт и символов Unicode.

## Что такое _«каналы»_?
Каналы (channels) – это логические (не физические) порталы, абстракции объектов более низкого уровня файловой системы (например, отображенные в памяти файлы и блокировки файлов), через которые осуществляется ввод/вывод данных, а буферы являются источниками или приёмниками этих переданных данных. При организации вывода, данные, которые необходимо отправить, помещаются в буфер, который затем передается в канал. При вводе, данные из канала помещаются в заранее предоставленный буфер.

Каналы напоминают трубопроводы, по которым эффективно транспортируются данные между буферами байтов и сущностями по ту сторону каналов. Каналы – это шлюзы, которые позволяют получить доступ к сервисам ввода/вывода операционной системы с минимальными накладными расходами, а буферы – внутренние конечные точки этих шлюзов, используемые для передачи и приема данных.

## Какие подклассы класса `InputStream` вы знаете, для чего они предназначены?
+ `InputStream` - абстрактный класс, описывающий поток ввода;
+ `BufferedInputStream` - буферизованный входной поток;
+ `ByteArrayInputStream` позволяет использовать буфер в памяти (массив байтов) в качестве источника данных для входного потока;
+ `DataInputStream` - входной поток для байтовых данных, включающий методы для чтения стандартных типов данных Java;
+ `FileInputStream` - входной поток для чтения информации из файла;
+ `FilterInputStream` - абстрактный класс, предоставляющий интерфейс для классов-надстроек, которые добавляют к существующим потокам полезные свойства;
+ `ObjectInputStream` - входной поток для объектов;
+ `StringBufferInputStream` превращает строку (`String`) во входной поток данных `InputStream`;
+ `PipedInputStream` реализует понятие входного канала;
+ `PushbackInputStream` - разновидность буферизации, обеспечивающая чтение байта с последующим его возвратом в поток, позволяет «заглянуть» во входной поток и увидеть, что оттуда поступит в следующий момент, не извлекая информации.
+ `SequenceInputStream` используется для слияния двух или более потоков `InputStream` в единый.

## Для чего используется `PushbackInputStream`?
Разновидность буферизации, обеспечивающая чтение байта с последующим его возвратом в поток. Класс `PushbackInputStream` представляет механизм «заглянуть» во входной поток и увидеть, что оттуда поступит в следующий момент, не извлекая информации.

У класса есть дополнительный метод unread().

## Для чего используется `SequenceInputStream`?
Класс `SequenceInputStream` позволяет сливать вместе несколько экземпляров класса `InputStream`. Конструктор принимает в качестве аргумента либо пару объектов класса `InputStream`, либо интерфейс `Enumeration`.

Во время работы класс выполняет запросы на чтение из первого объекта класса `InputStream` и до конца, а затем переключается на второй. При использовании интерфейса работа продолжится по всем объектам класса `InputStream`. По достижении конца, связанный с ним поток закрывается. Закрытие потока, созданного объектом класса `SequenceInputStream`, приводит к закрытию всех открытых потоков.

## Какой класс позволяет читать данные из входного байтового потока в формате примитивных типов данных?
Класс `DataInputStream` представляет поток ввода и предназначен для записи данных примитивных типов, таких, как `int`, `double` и т.д. Для каждого примитивного типа определен свой метод для считывания:

+ `boolean readBoolean()`: считывает из потока булевое однобайтовое значение
+ `byte readByte()`: считывает из потока 1 байт
+ `char readChar()`: считывает из потока значение `char`
+ `double readDouble()`: считывает из потока 8-байтовое значение `double`
+ `float readFloat()`: считывает из потока 4-байтовое значение `float`
+ `int readInt()`: считывает из потока целочисленное значение `int`
+ `long readLong()`: считывает из потока значение `long`
+ `short readShort()`: считывает значение `short`
+ `String readUTF()`: считывает из потока строку в кодировке UTF-8

## В чем отличие класса `PrintWriter` от `PrintStream`?
Прежде всего, в классе `PrintWriter` применен усовершенствованный способ работы с символами Unicode и другой механизм буферизации вывода: в классе PrintStream буфер вывода сбрасывался всякий раз, когда вызывался метод `print()` или `println()`, а при использовании класса `PrintWriter` существует возможность отказаться от автоматического сброса буферов, выполняя его явным образом при помощи метода `flush()`.

Кроме того, методы класса `PrintWriter` никогда не создают исключений. Для проверки ошибок необходимо явно вызвать метод `checkError()`.

+ `boolean isFile()`: возвращает значение `true`, если по указанному пути находится файл;
+ `boolean isHidden()`: возвращает значение `true`, если каталог или файл являются скрытыми;
+ `long length()`: возвращает размер файла в байтах;
+ `long lastModified()`: возвращает время последнего изменения файла или каталога;
+ `String[] list()`: возвращает массив файлов и подкаталогов, которые находятся в определенном каталоге;
+ `File[] listFiles()`: возвращает массив файлов и подкаталогов, которые находятся в определенном каталоге.

## Что вы знаете о `RandomAccessFile`?
Класс `java.io.RandomAccessFile` обеспечивает чтение и запись данных в произвольном месте файла. Он не является частью иерархии `InputStream` или `OutputStream`. Это полностью отдельный класс, имеющий свои собственные (в большинстве своем _native_) методы. Объяснением этого может быть то, что `RandomAccessFile` имеет во многом отличающееся поведение по сравнению с остальными классами ввода/вывода так как позволяет, в пределах файла, перемещаться вперед и назад.

`RandomAccessFile` имеет такие специфические методы как:

+ `getFilePointer()` для определения текущего местоположения в файле;
+ `seek()` для перемещения на новую позицию в файле;
+ `length()` для выяснения размера файла;
+ `setLength()` для установки размера файла;
+ `skipBytes()` для того, чтобы попытаться пропустить определённое число байт;
+ `getChannel()` для работы с уникальным файловым каналом, ассоциированным с заданным файлом;
+ методы для выполнения обычного и форматированного вывода из файла (`read()`, `readInt()`, `readLine()`, `readUTF()` и т.п.);
+ методы для обычной или форматированной записи в файл с прямым доступом (`write()`, `writeBoolean()`, `writeByte()` и т.п.).

Так же следует отметить, что конструкторы `RandomAccessFile` требуют второй аргумент, указывающий необходимый режим доступа к файлу - только чтение (`"r"`), чтение и запись (`"rw"`) или иную их разновидность.

## Какие режимы доступа к файлу есть у `RandomAccessFile`?
+ `"r"` открывает файл только для чтения. Запуск любых методов записи данных приведет к выбросу исключения `IOException`.
+ `"rw"` открывает файл для чтения и записи. Если файл еще не создан, то осуществляется попытка создать его.
+ `"rws"` открывает файл для чтения и записи подобно `"rw"`, но требует от системы при каждом изменении содержимого файла или метаданных синхронно записывать эти изменения на физический носитель.
+ `"rwd"` открывает файл для чтения и записи подобно `"rws"`, но требует от системы синхронно записывать изменения на физический носитель только при каждом изменении содержимого файла. Если изменяются метаданные, синхронная запись не требуется.

## Что такое _«абсолютный путь»_ и _«относительный путь»_?
__Абсолютный (полный) путь__ — это путь, который указывает на одно и то же место в файловой системе, вне зависимости от текущей рабочей директории или других обстоятельств. Полный путь всегда начинается с корневого каталога.

## Что такое _«символьная ссылка»_?
__Символьная (символическая) ссылка__ (также «симлинк», Symbolic link) — специальный файл в файловой системе, в котором, вместо пользовательских данных, содержится путь к файлу, который должен быть открыт при попытке обратиться к данной ссылке (файлу). Целью ссылки может быть любой объект: например, другая ссылка, файл, каталог или даже несуществующий файл (в последнем случае, при попытке открыть его, должно выдаваться сообщение об отсутствии файла).

Символьные ссылки используются для более удобной организации структуры файлов на компьютере, так как:

+ позволяют для одного файла или каталога иметь несколько имён и различных атрибутов;
+ свободны от некоторых ограничений, присущих жёстким ссылкам (последние действуют только в пределах одной файловой системы (одного раздела) и не могут ссылаться на каталоги).

## Что такое NIO.2 (Path, Files) и чем он отличается от java.io.File?
__NIO.2__ (пакет `java.nio.file`) — это API для работы с файловой системой, появившееся в Java 7 как замена устаревшему классу `java.io.File`. Центральные элементы NIO.2 — интерфейс `Path` и утилитный класс `Files`.

### Path vs File
`Path` — это интерфейс, представляющий путь в файловой системе. В отличие от `java.io.File`, объекты `Path` являются неизменяемыми (immutable) и обеспечивают более надёжную работу с путями.

```java
// Старый подход — java.io.File
File file = new File("/home/user/data.txt");

// Новый подход — java.nio.file.Path
Path path = Path.of("/home/user/data.txt");            // Java 11+
Path path2 = Paths.get("/home/user/data.txt");          // Java 7+
Path path3 = Path.of("/home", "user", "data.txt");      // составной путь
```

### Основные отличия Path от File

| Характеристика | `java.io.File` | `java.nio.file.Path` |
|---|---|---|
| Тип | Класс | Интерфейс |
| Иммутабельность | Нет | Да |
| Обработка ошибок | Возвращает `false`/`null` | Бросает информативные исключения (`NoSuchFileException`, `AccessDeniedException`) |
| Символьные ссылки | Ограниченная поддержка | Полная поддержка |
| Метаданные файлов | Ограниченный набор | Полный набор атрибутов через `Files.readAttributes()` |
| Обход дерева | Ручная рекурсия | `Files.walk()`, `Files.walkFileTree()` |

### Класс Files — утилитный класс
`Files` предоставляет набор статических методов для операций с файлами и каталогами:

```java
// Чтение и запись (Java 11+)
String content = Files.readString(Path.of("file.txt"));
Files.writeString(Path.of("output.txt"), "Содержимое", StandardCharsets.UTF_8);

// Чтение и запись (Java 7+)
byte[] bytes = Files.readAllBytes(Path.of("file.txt"));
List<String> lines = Files.readAllLines(Path.of("file.txt"), StandardCharsets.UTF_8);
Files.write(Path.of("output.txt"), lines, StandardCharsets.UTF_8);

// Ленивое чтение строк (возвращает Stream)
try (Stream<String> stream = Files.lines(Path.of("large_file.txt"))) {
    stream.filter(line -> line.contains("ERROR"))
          .forEach(System.out::println);
}

// Обход дерева каталогов
try (Stream<Path> paths = Files.walk(Path.of("/home/user"))) {
    paths.filter(Files::isRegularFile)
         .filter(p -> p.toString().endsWith(".java"))
         .forEach(System.out::println);
}

// Копирование, перемещение, удаление
Files.copy(source, target, StandardCopyOption.REPLACE_EXISTING);
Files.move(source, target, StandardCopyOption.ATOMIC_MOVE);
Files.delete(path);               // бросает исключение если файл не найден
Files.deleteIfExists(path);       // возвращает false если файл не найден
```

### Преобразование между File и Path
```java
// File -> Path
File file = new File("data.txt");
Path path = file.toPath();

// Path -> File
Path path = Path.of("data.txt");
File file = path.toFile();
```

### Важное
+ `Path` — это интерфейс (не класс), что позволяет работать с разными файловыми системами (например, ZIP через `FileSystem`).
+ `Files.readString()` и `Files.writeString()` появились в Java 11 и являются самым лаконичным способом работы с текстовыми файлами.
+ Методы `Files`, возвращающие `Stream` (`lines()`, `walk()`, `find()`), **обязательно** нужно закрывать — используйте `try-with-resources`.
+ `Path.of()` — предпочтительный способ создания `Path` начиная с Java 11 (вместо `Paths.get()`).

### Частые ошибки
+ Забывают закрывать `Stream`, возвращаемый `Files.lines()` и `Files.walk()` — это приводит к утечке файловых дескрипторов.
+ Используют `Files.readAllLines()` для огромных файлов — весь файл загружается в память. Для больших файлов следует использовать `Files.lines()`.
+ Не указывают `StandardCopyOption.REPLACE_EXISTING` при копировании — получают `FileAlreadyExistsException`.
+ Путают `Files.delete()` (бросает исключение) и `Files.deleteIfExists()` (возвращает boolean).

### Как используется в 2026
+ NIO.2 является стандартом де-факто для работы с файловой системой. Использование `java.io.File` в новом коде считается устаревшей практикой.
+ В Spring Boot и других фреймворках `Path` и `Files` используются повсеместно для работы с ресурсами, конфигурацией и загрузкой файлов.
+ В сочетании с виртуальными потоками (Project Loom, Java 21+) блокирующие операции `Files` становятся более эффективными без необходимости перехода на асинхронный ввод/вывод.

## Как работать с файлами и каталогами через Files API?
Класс `java.nio.file.Files` предоставляет исчерпывающий набор статических методов для работы с файлами и каталогами. Ниже рассмотрены основные практические сценарии.

### Чтение и запись файлов
```java
Path path = Path.of("example.txt");

// Запись строки в файл (Java 11+)
Files.writeString(path, "Привет, мир!\nВторая строка.", StandardCharsets.UTF_8);

// Запись списка строк
Files.write(path, List.of("строка 1", "строка 2", "строка 3"), StandardCharsets.UTF_8);

// Дозапись в файл
Files.writeString(path, "\nДополнительная строка",
    StandardCharsets.UTF_8, StandardOpenOption.APPEND);

// Чтение всего содержимого как строки (Java 11+)
String content = Files.readString(path, StandardCharsets.UTF_8);

// Чтение построчно в список
List<String> lines = Files.readAllLines(path, StandardCharsets.UTF_8);

// Чтение в байтовый массив
byte[] data = Files.readAllBytes(path);
```

### Копирование, перемещение и удаление
```java
Path source = Path.of("source.txt");
Path target = Path.of("backup/source_copy.txt");

// Копирование с заменой существующего файла
Files.copy(source, target, StandardCopyOption.REPLACE_EXISTING);

// Копирование с сохранением атрибутов
Files.copy(source, target,
    StandardCopyOption.REPLACE_EXISTING,
    StandardCopyOption.COPY_ATTRIBUTES);

// Копирование из InputStream в файл
try (InputStream is = new URL("https://example.com/file.txt").openStream()) {
    Files.copy(is, target, StandardCopyOption.REPLACE_EXISTING);
}

// Перемещение (атомарное, если файловая система поддерживает)
Files.move(source, target, StandardCopyOption.ATOMIC_MOVE);

// Удаление
Files.delete(path);              // бросает NoSuchFileException если не существует
Files.deleteIfExists(path);      // возвращает false если не существует
```

### Создание каталогов и файлов
```java
// Создание одного каталога (родитель должен существовать)
Files.createDirectory(Path.of("new_dir"));

// Создание вложенных каталогов (аналог mkdir -p)
Files.createDirectories(Path.of("a/b/c/d"));

// Создание временного файла
Path tempFile = Files.createTempFile("prefix_", ".tmp");

// Создание временного каталога
Path tempDir = Files.createTempDirectory("my_temp_");
```

### Получение содержимого каталога
```java
// Листинг файлов в каталоге (только первый уровень)
try (DirectoryStream<Path> stream = Files.newDirectoryStream(Path.of("src"))) {
    for (Path entry : stream) {
        System.out.println(entry.getFileName());
    }
}

// С фильтром по расширению
try (DirectoryStream<Path> stream =
         Files.newDirectoryStream(Path.of("src"), "*.java")) {
    for (Path entry : stream) {
        System.out.println(entry);
    }
}

// Через Stream API (Java 8+)
try (Stream<Path> stream = Files.list(Path.of("src"))) {
    stream.filter(Files::isRegularFile)
          .forEach(System.out::println);
}
```

### Рекурсивный обход дерева каталогов
```java
// Files.walk — рекурсивный обход с возвратом Stream<Path>
try (Stream<Path> stream = Files.walk(Path.of("/project/src"))) {
    List<Path> javaFiles = stream
        .filter(Files::isRegularFile)
        .filter(p -> p.toString().endsWith(".java"))
        .collect(Collectors.toList());
    javaFiles.forEach(System.out::println);
}

// Files.walk с ограничением глубины
try (Stream<Path> stream = Files.walk(Path.of("/project"), 2)) {
    stream.forEach(System.out::println);
}

// Files.find — обход с фильтром по атрибутам
try (Stream<Path> stream = Files.find(Path.of("/project/src"), Integer.MAX_VALUE,
        (path, attrs) -> attrs.isRegularFile() && path.toString().endsWith(".java"))) {
    stream.forEach(System.out::println);
}
```

### FileVisitor — продвинутый обход дерева
```java
// Пример: рекурсивное удаление каталога
Files.walkFileTree(Path.of("dir_to_delete"), new SimpleFileVisitor<>() {
    @Override
    public FileVisitResult visitFile(Path file, BasicFileAttributes attrs)
            throws IOException {
        Files.delete(file);
        return FileVisitResult.CONTINUE;
    }

    @Override
    public FileVisitResult postVisitDirectory(Path dir, IOException exc)
            throws IOException {
        Files.delete(dir);
        return FileVisitResult.CONTINUE;
    }
});
```

### Проверка свойств файлов
```java
Path path = Path.of("data.txt");

boolean exists = Files.exists(path);
boolean isFile = Files.isRegularFile(path);
boolean isDir = Files.isDirectory(path);
boolean isLink = Files.isSymbolicLink(path);
boolean readable = Files.isReadable(path);
boolean writable = Files.isWritable(path);
long size = Files.size(path);
FileTime modified = Files.getLastModifiedTime(path);
String mimeType = Files.probeContentType(path);    // например "text/plain"
```

### Важное
+ Все методы `Files`, возвращающие `Stream` (`list()`, `walk()`, `find()`, `lines()`), открывают ресурсы файловой системы и требуют `try-with-resources`.
+ `Files.createDirectories()` не бросает исключение, если каталог уже существует (в отличие от `Files.createDirectory()`).
+ `Files.walk()` по умолчанию не следует символьным ссылкам. Для следования используйте `FileVisitOption.FOLLOW_LINKS`.
+ `DirectoryStream` — это не `Stream` из Java 8, а `Iterable`. Для Stream API используйте `Files.list()`.

### Частые ошибки
+ Используют `Files.walk()` без `try-with-resources` — утечка файловых дескрипторов на больших деревьях каталогов.
+ Пытаются удалить непустой каталог через `Files.delete()` — получают `DirectoryNotEmptyException`. Необходимо сначала удалить содержимое (через `walkFileTree`).
+ Забывают проверять `Files.exists()` перед операциями и не обрабатывают `NoSuchFileException`.
+ Используют `Files.list()` рекурсивно вместо `Files.walk()` — получают запутанный код и пропуск вложенных каталогов.

### Как используется в 2026
+ `Files API` является стандартным инструментом для всех операций с файловой системой в Java-проектах.
+ В микросервисной архитектуре `Files.createTempFile()` и `Files.createTempDirectory()` активно используются для обработки загружаемых файлов и промежуточных данных.
+ Фреймворки (Spring, Quarkus) используют `Files.walk()` и `Files.find()` для сканирования classpath и автоматического обнаружения ресурсов.
+ С Java 21+ виртуальные потоки делают блокирующие вызовы `Files` столь же масштабируемыми, как и асинхронные альтернативы.

## Что такое ByteBuffer и как с ним работать?
`ByteBuffer` — это буфер для хранения байтовых данных, являющийся ключевым компонентом пакета `java.nio`. Он используется для эффективного чтения и записи данных через каналы (channels) NIO.

### Концепция буфера
Буфер — это линейная конечная последовательность элементов определённого примитивного типа. Помимо содержимого, буфер имеет четыре ключевых свойства:

+ **capacity** (ёмкость) — максимальное количество элементов в буфере. Задаётся при создании и не изменяется.
+ **position** (позиция) — индекс следующего элемента для чтения или записи. Начинается с 0.
+ **limit** (предел) — индекс первого элемента, который **не должен** читаться или записываться.
+ **mark** (метка) — сохранённая позиция, к которой можно вернуться через `reset()`.

Инвариант: `0 <= mark <= position <= limit <= capacity`

### Создание ByteBuffer
```java
// Выделение в куче (heap buffer)
ByteBuffer heapBuffer = ByteBuffer.allocate(1024);

// Выделение в нативной памяти (direct buffer)
ByteBuffer directBuffer = ByteBuffer.allocateDirect(1024);

// Обёртка над существующим массивом
byte[] array = new byte[1024];
ByteBuffer wrappedBuffer = ByteBuffer.wrap(array);

// Обёртка с указанием offset и length
ByteBuffer partialWrap = ByteBuffer.wrap(array, 10, 100);
```

**Heap buffer** vs **Direct buffer**:
+ Heap-буфер хранится в куче JVM, быстрее создаётся и управляется сборщиком мусора.
+ Direct-буфер выделяется в нативной памяти ОС, медленнее создаётся, но обеспечивает более эффективный ввод/вывод (нет дополнительного копирования данных при взаимодействии с ОС).
+ Direct-буферы следует использовать для долгоживущих объектов с интенсивным вводом/выводом.

### Основные операции
```java
ByteBuffer buffer = ByteBuffer.allocate(1024);

// === Запись данных в буфер ===
buffer.put((byte) 65);                // записать один байт
buffer.putInt(42);                     // записать int (4 байта)
buffer.putLong(123456789L);            // записать long (8 байт)
buffer.put("Hello".getBytes(StandardCharsets.UTF_8));  // записать массив байт

// === Переключение из режима записи в режим чтения ===
buffer.flip();
// flip() устанавливает limit = position, position = 0

// === Чтение данных из буфера ===
byte b = buffer.get();                 // прочитать один байт
int num = buffer.getInt();             // прочитать int
long lng = buffer.getLong();           // прочитать long

// === Подготовка буфера к повторной записи ===
buffer.clear();
// clear() устанавливает position = 0, limit = capacity
// Данные не стираются, но будут перезаписаны

// === Компактирование: сохранение непрочитанных данных ===
buffer.compact();
// compact() копирует непрочитанные данные в начало буфера,
// position = количество оставшихся данных, limit = capacity
```

### Паттерн чтения из канала
```java
try (FileChannel channel = FileChannel.open(Path.of("data.bin"),
        StandardOpenOption.READ)) {
    ByteBuffer buffer = ByteBuffer.allocate(4096);

    while (channel.read(buffer) != -1) {
        buffer.flip();             // переключение в режим чтения
        while (buffer.hasRemaining()) {
            byte b = buffer.get(); // обработка данных
        }
        buffer.clear();            // подготовка к следующему чтению
    }
}
```

### Паттерн записи в канал
```java
try (FileChannel channel = FileChannel.open(Path.of("output.bin"),
        StandardOpenOption.CREATE, StandardOpenOption.WRITE)) {
    ByteBuffer buffer = ByteBuffer.allocate(4096);

    buffer.putInt(42);
    buffer.put("Данные".getBytes(StandardCharsets.UTF_8));

    buffer.flip();                 // переключение в режим чтения для канала
    while (buffer.hasRemaining()) {
        channel.write(buffer);
    }
}
```

### Полезные методы
```java
ByteBuffer buffer = ByteBuffer.allocate(100);

buffer.remaining();       // количество элементов между position и limit
buffer.hasRemaining();    // есть ли элементы между position и limit
buffer.rewind();          // position = 0, mark сбрасывается (перечитать данные)
buffer.mark();            // сохранить текущую позицию
buffer.reset();           // вернуться к сохранённой позиции (mark)
buffer.duplicate();       // создать дубликат буфера (общий массив, разные позиции)
buffer.slice();           // создать срез от position до limit
buffer.order(ByteOrder.LITTLE_ENDIAN);  // задать порядок байт
```

### Важное
+ Вызов `flip()` обязателен при переключении из режима записи в режим чтения. Без него `position` будет указывать на конец записанных данных, а не на начало.
+ `clear()` не стирает данные — он лишь сбрасывает позиции. Для реального обнуления используйте цикл с `put((byte) 0)`.
+ `ByteBuffer` поддерживает все примитивные типы через методы `putInt()`, `putLong()`, `putDouble()` и т.д.
+ По умолчанию байтовый порядок — `BIG_ENDIAN` (сетевой порядок). Для работы с данными в нативном порядке используйте `ByteOrder.nativeOrder()`.

### Частые ошибки
+ Забывают вызвать `flip()` после записи перед чтением — читают «мусор» или не читают ничего.
+ Путают `clear()` и `compact()`: `clear()` отбрасывает все данные, `compact()` сохраняет непрочитанные.
+ Создают direct-буферы для мелких короткоживущих операций — это неэффективно из-за дорогого выделения нативной памяти.
+ Не учитывают, что `channel.write(buffer)` может записать не все данные за один вызов — необходим цикл с `hasRemaining()`.

### Как используется в 2026
+ `ByteBuffer` является фундаментом для высокопроизводительного ввода/вывода в Netty, Apache MINA, Vert.x и других сетевых фреймворках.
+ Direct-буферы используются в связке с Panama Foreign Memory API (JEP 454, Java 22) для взаимодействия с нативным кодом без JNI.
+ В протоколах сериализации (Protocol Buffers, FlatBuffers, Apache Arrow) `ByteBuffer` используется для zero-copy доступа к данным.
+ С виртуальными потоками (Java 21+) паттерны работы с `ByteBuffer` упрощаются, так как блокирующие операции не блокируют потоки ОС.

## Как реализовать чтение большого файла без загрузки в память?
При работе с большими файлами (сотни мегабайт и более) важно не загружать всё содержимое в оперативную память. Java предоставляет несколько подходов для потокового и ленивого чтения.

### 1. Files.lines() — ленивый Stream строк
Самый простой и идиоматичный способ для текстовых файлов. Возвращает `Stream<String>`, который читает строки лениво (по мере необходимости):

```java
try (Stream<String> lines = Files.lines(Path.of("huge_file.txt"), StandardCharsets.UTF_8)) {
    long errorCount = lines
        .filter(line -> line.contains("ERROR"))
        .count();
    System.out.println("Количество ошибок: " + errorCount);
}
```

### 2. BufferedReader.lines() — ленивый Stream из BufferedReader
Аналогичный подход, но с возможностью задать размер буфера:

```java
try (BufferedReader reader = Files.newBufferedReader(Path.of("huge_file.txt"),
        StandardCharsets.UTF_8)) {
    reader.lines()
          .filter(line -> line.startsWith("WARN"))
          .limit(100)
          .forEach(System.out::println);
}
```

### 3. BufferedReader с ручным чтением строк
Классический подход, даёт полный контроль:

```java
try (BufferedReader reader = Files.newBufferedReader(Path.of("huge_file.txt"))) {
    String line;
    int lineNumber = 0;
    while ((line = reader.readLine()) != null) {
        lineNumber++;
        if (line.contains("критическая ошибка")) {
            System.out.printf("Строка %d: %s%n", lineNumber, line);
        }
    }
}
```

### 4. NIO Channel с ByteBuffer — для бинарных файлов
Эффективный подход для бинарных данных:

```java
try (FileChannel channel = FileChannel.open(Path.of("huge_binary.dat"),
        StandardOpenOption.READ)) {
    ByteBuffer buffer = ByteBuffer.allocateDirect(8192); // 8 КБ буфер
    long totalBytes = 0;

    while (channel.read(buffer) != -1) {
        buffer.flip();
        while (buffer.hasRemaining()) {
            buffer.get(); // обработка байта
            totalBytes++;
        }
        buffer.clear();
    }
    System.out.println("Обработано байт: " + totalBytes);
}
```

### 5. Memory-mapped файлы (MappedByteBuffer)
Файл отображается напрямую в виртуальную память процесса. Операционная система сама управляет загрузкой и выгрузкой страниц:

```java
try (FileChannel channel = FileChannel.open(Path.of("huge_file.dat"),
        StandardOpenOption.READ)) {
    long fileSize = channel.size();

    // Отображаем файл целиком (или по частям для файлов > 2 ГБ)
    MappedByteBuffer mappedBuffer = channel.map(
        FileChannel.MapMode.READ_ONLY, 0, fileSize);

    // Чтение данных — ОС подгружает страницы по мере обращения
    while (mappedBuffer.hasRemaining()) {
        byte b = mappedBuffer.get();
        // обработка
    }
}

// Для файлов больше Integer.MAX_VALUE — чтение по частям
try (FileChannel channel = FileChannel.open(Path.of("very_large.dat"),
        StandardOpenOption.READ)) {
    long fileSize = channel.size();
    long position = 0;
    int chunkSize = 256 * 1024 * 1024; // 256 МБ

    while (position < fileSize) {
        long remaining = fileSize - position;
        int mapSize = (int) Math.min(chunkSize, remaining);
        MappedByteBuffer buffer = channel.map(
            FileChannel.MapMode.READ_ONLY, position, mapSize);

        while (buffer.hasRemaining()) {
            buffer.get(); // обработка
        }
        position += mapSize;
    }
}
```

### 6. Scanner — для парсинга структурированных данных
```java
try (Scanner scanner = new Scanner(Path.of("data.csv"), StandardCharsets.UTF_8)) {
    scanner.useDelimiter(",|\\n");
    while (scanner.hasNext()) {
        String token = scanner.next();
        // обработка токена
    }
}
```

### Сравнение подходов

| Подход | Тип данных | Потребление памяти | Скорость | Удобство |
|---|---|---|---|---|
| `Files.lines()` | Текст | Низкое | Средняя | Высокое |
| `BufferedReader` | Текст | Низкое | Средняя | Среднее |
| `FileChannel + ByteBuffer` | Бинарные | Низкое (размер буфера) | Высокая | Низкое |
| `MappedByteBuffer` | Бинарные | Управляется ОС | Очень высокая | Среднее |
| `Scanner` | Структурированный текст | Низкое | Низкая | Высокое |

### Важное
+ `Files.lines()` — это ленивый `Stream`. Строки читаются по мере потребления, а не все сразу.
+ `MappedByteBuffer` не загружает весь файл в кучу JVM — он отображается в виртуальную память и ОС подгружает страницы по требованию.
+ Для файлов больше 2 ГБ `MappedByteBuffer` нужно создавать по частям, так как `map()` принимает `int size`.
+ Размер буфера `ByteBuffer` влияет на производительность: слишком маленький буфер — много системных вызовов, слишком большой — неэффективное использование памяти. Обычно оптимален размер 8-64 КБ.

### Частые ошибки
+ Используют `Files.readAllBytes()` или `Files.readAllLines()` для больших файлов — приводит к `OutOfMemoryError`.
+ Забывают закрывать `Stream<String>` от `Files.lines()` — утечка файловых дескрипторов.
+ Не используют `try-with-resources` при работе с `FileChannel` — незакрытые каналы приводят к блокировке файлов.
+ Применяют `MappedByteBuffer` для мелких файлов — накладные расходы на отображение в память перевешивают выгоду.

### Как используется в 2026
+ `Files.lines()` в сочетании со Stream API — основной подход для обработки текстовых логов и CSV-файлов.
+ Memory-mapped файлы используются в базах данных (Apache Lucene, RocksDB), кэшах и высокопроизводительных системах обмена сообщениями (Chronicle Queue).
+ С появлением Foreign Memory API (Java 22+) `MemorySegment` постепенно заменяет `MappedByteBuffer` для низкоуровневой работы с памятью, обеспечивая безопасное управление жизненным циклом.
+ В big data экосистеме (Apache Spark, Flink) потоковое чтение файлов является стандартным паттерном для обработки данных, не помещающихся в память.

## Что такое try-with-resources и как правильно закрывать ресурсы?
`try-with-resources` — это конструкция языка Java (появилась в Java 7), которая автоматически закрывает ресурсы, реализующие интерфейс `AutoCloseable`, при выходе из блока `try`. Это устраняет необходимость вручную закрывать потоки, соединения и другие ресурсы в блоке `finally`.

### Интерфейс AutoCloseable
```java
public interface AutoCloseable {
    void close() throws Exception;
}

// Closeable extends AutoCloseable (с ограничением IOException)
public interface Closeable extends AutoCloseable {
    void close() throws IOException;
}
```

Все потоки ввода/вывода (`InputStream`, `OutputStream`, `Reader`, `Writer`), каналы (`Channel`), соединения с БД (`Connection`, `Statement`, `ResultSet`) реализуют `AutoCloseable` или `Closeable`.

### Базовый синтаксис
```java
// Старый подход (до Java 7) — многословный и подверженный ошибкам
InputStream is = null;
try {
    is = new FileInputStream("file.txt");
    // чтение данных
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (is != null) {
        try {
            is.close();
        } catch (IOException e) {
            e.printStackTrace(); // исключение при закрытии может «поглотить» оригинальное
        }
    }
}

// Новый подход (Java 7+) — лаконичный и безопасный
try (InputStream is = new FileInputStream("file.txt")) {
    // чтение данных
} catch (IOException e) {
    e.printStackTrace();
}
// is.close() вызывается автоматически
```

### Несколько ресурсов
Можно объявить несколько ресурсов через точку с запятой. Ресурсы закрываются в **обратном** порядке объявления:

```java
try (FileInputStream fis = new FileInputStream("input.txt");
     BufferedInputStream bis = new BufferedInputStream(fis);
     FileOutputStream fos = new FileOutputStream("output.txt");
     BufferedOutputStream bos = new BufferedOutputStream(fos)) {

    byte[] buffer = new byte[1024];
    int bytesRead;
    while ((bytesRead = bis.read(buffer)) != -1) {
        bos.write(buffer, 0, bytesRead);
    }
}
// Порядок закрытия: bos -> fos -> bis -> fis
```

### Effectively final переменные (Java 9+)
Начиная с Java 9, в `try-with-resources` можно использовать переменные, объявленные вне блока, если они effectively final:

```java
// Java 9+
BufferedReader reader = new BufferedReader(new FileReader("file.txt"));
try (reader) {  // reader — effectively final
    String line = reader.readLine();
    System.out.println(line);
}
// reader.close() вызывается автоматически
```

### Подавленные исключения (Suppressed Exceptions)
Если исключение возникает и в блоке `try`, и в методе `close()`, исключение из `close()` **подавляется** и добавляется к основному исключению:

```java
public class ProblematicResource implements AutoCloseable {
    public void doWork() {
        throw new RuntimeException("Ошибка в doWork()");
    }

    @Override
    public void close() {
        throw new RuntimeException("Ошибка в close()");
    }
}

try (ProblematicResource resource = new ProblematicResource()) {
    resource.doWork();
} catch (RuntimeException e) {
    System.out.println("Основное: " + e.getMessage());
    // Основное: Ошибка в doWork()

    for (Throwable suppressed : e.getSuppressed()) {
        System.out.println("Подавленное: " + suppressed.getMessage());
        // Подавленное: Ошибка в close()
    }
}
```

### Создание собственного AutoCloseable ресурса
```java
public class DatabaseConnection implements AutoCloseable {
    private final Connection connection;

    public DatabaseConnection(String url) throws SQLException {
        this.connection = DriverManager.getConnection(url);
        System.out.println("Соединение открыто");
    }

    public ResultSet query(String sql) throws SQLException {
        return connection.createStatement().executeQuery(sql);
    }

    @Override
    public void close() throws SQLException {
        if (connection != null && !connection.isClosed()) {
            connection.close();
            System.out.println("Соединение закрыто");
        }
    }
}

// Использование
try (DatabaseConnection db = new DatabaseConnection("jdbc:h2:mem:test")) {
    ResultSet rs = db.query("SELECT * FROM users");
    // обработка результатов
}
// close() вызывается автоматически
```

### Практические примеры с потоками ввода/вывода
```java
// Копирование файла
try (InputStream in = Files.newInputStream(Path.of("source.txt"));
     OutputStream out = Files.newOutputStream(Path.of("target.txt"))) {
    in.transferTo(out);  // Java 9+
}

// Построчная обработка с BufferedReader
try (BufferedReader reader = Files.newBufferedReader(Path.of("log.txt"))) {
    reader.lines()
          .filter(line -> line.contains("ERROR"))
          .forEach(System.out::println);
}

// Работа с NIO каналами
try (FileChannel src = FileChannel.open(Path.of("source.dat"), StandardOpenOption.READ);
     FileChannel dst = FileChannel.open(Path.of("dest.dat"),
         StandardOpenOption.CREATE, StandardOpenOption.WRITE)) {
    src.transferTo(0, src.size(), dst);
}

// Stream из Files.lines() — тоже требует закрытия
try (Stream<String> lines = Files.lines(Path.of("data.txt"))) {
    Optional<String> first = lines.filter(l -> l.startsWith("ID:")).findFirst();
    first.ifPresent(System.out::println);
}
```

### Важное
+ `close()` вызывается **до** блоков `catch` и `finally`, но ресурс доступен в этих блоках как effectively final переменная (Java 9+).
+ Ресурсы закрываются в **обратном** порядке объявления — это важно при наличии зависимостей между ресурсами.
+ Если `close()` бросает исключение, а в `try` тоже было исключение, исключение из `close()` становится suppressed.
+ `try-with-resources` работает только с объектами, реализующими `AutoCloseable` (или `Closeable`).
+ `Stream`, возвращённые из `Files.lines()`, `Files.walk()`, `Files.list()`, тоже реализуют `AutoCloseable` и должны закрываться.

### Частые ошибки
+ Закрывают ресурсы в неправильном порядке в `finally` блоке — это была основная причина появления `try-with-resources`.
+ Забывают, что `close()` может бросить исключение, которое «поглотит» оригинальное исключение (в старом подходе через `finally`).
+ Используют `try-with-resources` без `catch` и думают, что исключения обрабатываются — `try-with-resources` только закрывает ресурсы, но не обрабатывает исключения.
+ Оборачивают потоки друг в друга и закрывают оба отдельно — при закрытии обёртки (например, `BufferedInputStream`) внутренний поток тоже закрывается, повторное закрытие может вызвать проблемы.
+ Не используют `try-with-resources` для `Stream` от `Files.lines()` — забывая, что этот `Stream` удерживает открытый файловый дескриптор.

### Как используется в 2026
+ `try-with-resources` является обязательным стандартом во всех Java-проектах. Код с ручным закрытием в `finally` считается устаревшей практикой.
+ Статические анализаторы (SonarQube, SpotBugs, IntelliJ IDEA) автоматически обнаруживают незакрытые ресурсы и предлагают рефакторинг на `try-with-resources`.
+ В Project Loom (Java 21+) `StructuredTaskScope` реализует `AutoCloseable`, позволяя управлять жизненным циклом параллельных задач через `try-with-resources`.
+ Все современные фреймворки (Spring, Micronaut, Quarkus) активно используют `AutoCloseable` для управления ресурсами: соединения с БД, HTTP-клиенты, потоки данных.
